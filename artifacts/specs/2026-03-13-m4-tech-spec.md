# M4 Final Tech Spec — Real-time GPS Pipeline + PostgreSQL + VRP-TW Prototype

**Date**: 2026-03-13

## Problem Statement
The driver app pushes GPS and the parent app listens via WebSocket, but the backend WebSocket endpoint lacks authentication and the GPS buffer is never flushed to PostgreSQL. Additionally, Alembic migrations have never been verified against a real PostgreSQL instance. The VRP-TW routing engine is a stub.

## Goals
1. Harden WebSocket GPS endpoint with JWT auth
2. Wire GPS buffer flush as a background task
3. Verify Alembic migrations on real PostgreSQL
4. Build VRP-TW prototype with OR-Tools (parallel, non-blocking)

## Non-goals
- Admin dashboard
- Background GPS tracking on mobile
- Production-grade VRP-TW with Kakao API distance matrix

## Track A: WebSocket GPS Hardening

### A1. JWT Authentication for WebSocket
WebSocket connections don't support HTTP headers in browser. Use query parameter approach:

```
ws://host/api/v1/telemetry/ws/vehicles/{vehicle_id}?token=<JWT>
```

Implementation in `vehicle_telemetry/router.py`:
- Extract `token` from `websocket.query_params`
- Decode JWT using existing `decode_token()` from auth service
- Verify token type is `access` and user exists + is active
- On invalid token: `await websocket.close(code=4001, reason="Unauthorized")`
- Requires DB session for user lookup — use `async_session_factory` directly (WebSocket handlers can't use Depends for DB in the same way)

### A2. Message Format Verification
Current `update_gps()` publishes:
```json
{
  "vehicle_id": "uuid-string",
  "latitude": 37.1234,
  "longitude": 127.5678,
  "heading": 90.0,
  "speed": 60.0,
  "recorded_at": "2026-03-13T10:00:00+00:00"
}
```
Mobile `GpsLocation` type expects:
```typescript
{ vehicle_id: string, latitude: number, longitude: number, heading: number|null, speed: number|null, recorded_at: string }
```
**These already match.** No changes needed.

### A3. Connection Lifecycle Improvements
- Add ping/pong keepalive: send WebSocket ping every 30s
- Handle `asyncio.CancelledError` in addition to `WebSocketDisconnect`
- Log connection/disconnection events

## Track B: GPS Buffer Flush Background Task

### B1. Lifespan Background Task
Add to `main.py` lifespan:
- On startup, launch `asyncio.create_task(gps_flush_loop())`
- `gps_flush_loop()` runs every 30s:
  1. Scan Redis for `gps_buffer:*` keys using `KEYS` pattern (or maintain a set of active vehicle IDs)
  2. For each buffer key, call `flush_gps_buffer(redis, db, vehicle_id)`
  3. Log flush counts
- On shutdown, cancel the task

### B2. Active Vehicle Tracking
Instead of scanning keys (expensive), maintain a Redis SET `active_vehicles`:
- `update_gps()` adds vehicle_id to set with `SADD`
- `gps_flush_loop()` reads `SMEMBERS active_vehicles`
- After flush, if buffer is empty and GPS key has expired (vehicle inactive >5min), remove from set

## Track D: VRP-TW Prototype (Parallel)

### D1. Add OR-Tools Dependency
Add `ortools>=9.9.0` to `pyproject.toml` dependencies.

### D2. Solver Implementation
File: `backend/app/modules/routing_engine/solver.py`

Input model:
```python
@dataclass
class Stop:
    id: str          # schedule instance ID
    latitude: float
    longitude: float
    time_window: tuple[int, int]  # minutes from midnight (e.g., 780=1PM, 1080=6PM)
    academy_id: str  # destination academy

@dataclass
class VehicleSpec:
    id: str
    capacity: int
    depot_lat: float   # academy location (start/end point)
    depot_lng: float

@dataclass
class RouteSolution:
    vehicle_id: str
    ordered_stops: list[str]  # stop IDs in visit order
    total_distance_km: float
    total_duration_min: float
```

Algorithm:
1. Build Euclidean distance matrix between all stops + depot (academy)
2. Create OR-Tools routing model with:
   - Capacity constraints per vehicle
   - Time window constraints per stop
   - Minimize total distance
3. Solve with time limit (30s per SRS)
4. Return ordered stop list per vehicle

### D3. Service Layer
File: `backend/app/modules/routing_engine/service.py`
- `generate_route_plan(db, academy_id, plan_date)`:
  1. Query DailyScheduleInstance for the academy + date (status=scheduled)
  2. Query Vehicles assigned for the date
  3. Build stops + vehicle specs
  4. Call solver
  5. Persist RoutePlan rows

### D4. API Endpoint
- `POST /api/v1/routes/generate` — triggers route generation for an academy + date
- `GET /api/v1/routes/{vehicle_id}/{date}` — returns the route plan for driver app

### D5. Test with Synthetic Data
Unit test with 10-15 stops, 2-3 vehicles, verify solver produces valid routes within 30s.
