# Milestone 4 Report — Real-time GPS Pipeline + PostgreSQL + VRP-TW Prototype

**Date**: 2026-03-13
**Status**: COMPLETE

## Scope
Close the end-to-end real-time loop: driver GPS → backend WebSocket → parent app. Verify PostgreSQL schema. Build VRP-TW routing prototype.

## What Was Delivered

### 1. WebSocket GPS Endpoint Hardening
- JWT authentication via `?token=` query parameter
- User existence + active status verification against DB
- Graceful close with code 4001 on auth failure
- Ping/pong keepalive every 30s
- Connection/disconnection logging

### 2. GPS Buffer Flush Background Task
- `gps_flush_loop()` runs every 30s as an asyncio background task in FastAPI lifespan
- Reads `active_vehicles` Redis SET (populated by `update_gps()` via SADD)
- Flushes each vehicle's GPS buffer to `gps_history` PostgreSQL table
- Auto-removes inactive vehicles (no GPS data + expired latest position)
- Graceful shutdown via task cancellation

### 3. PostgreSQL Migration Verification
- `alembic upgrade head` executed successfully on real PostgreSQL 16
- 14 tables created with correct schema (users, vehicles, gps_history, etc.)
- Health check confirms: `{"status":"ok","redis":"connected","database":"connected"}`

### 4. VRP-TW Route Optimization Prototype
- OR-Tools constraint solver with:
  - Euclidean distance matrix (Korea lat/lng approximation)
  - Vehicle capacity constraints
  - Time window constraints per stop (±15 min from pickup time)
  - Guided Local Search metaheuristic
  - 30s time limit (per SRS requirement)
- Service layer: `generate_route_plan()` bridges DB → solver → `RoutePlan` persistence
- API endpoints:
  - `POST /api/v1/routes/generate` — trigger route generation (admin only)
  - `GET /api/v1/routes/{vehicle_id}/{date}` — retrieve route plan (driver/admin)
