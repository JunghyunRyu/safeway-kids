## P21. 대차 기사 — 최동훈 (30세, 대리운전 경력만 있음)

### 상황 배경

정규 기사 이상철(55세)이 오전 7:30에 갑자기 병가 통보. 학원 원장이 급히 대체 기사 최동훈을 호출.
첫 픽업은 오전 8:30. 최동훈은 앱을 처음 본다.
학원 원장의 안내: "앱 깔고 로그인하면 다 나와요."

### 시간순 시뮬레이션

**07:50 — 앱 설치 시도**
- Play Store에서 "세이프웨이 키즈" 검색, 설치 (약 2분)
- 앱 실행 → `LoginScreen.tsx` 표시

**07:52 — 로그인 시도**
- 로그인 화면에 역할 선택 그리드 표시: 학부모 / 기사 / 안전도우미 / 관리자
- "기사" 버튼을 눌러 역할 선택
- 전화번호, 이름 입력 후 로그인
- **빈틈 1:** 현재 `LoginScreen`은 "개발 모드 로그인"이라고 표시됨. 프로덕션에서는 OTP 인증 플로우가 적용될 것이나, **대차 기사의 계정이 사전에 시스템에 등록되어 있지 않으면 로그인 자체가 불가능**. 학원 관리자가 웹 대시보드에서 기사 계정을 생성하고 차량 배정을 해줘야 하는데, 이 과정이 30분 안에 완료 가능한지 불명확
- **빈틈 2:** 학원 원장이 "대차 기사 등록 → 차량 배정 변경"을 한 번도 해본 적 없을 가능성 높음. 긴급 대차 시나리오에 대한 관리자 가이드 부재

**07:55 — 로그인 성공 가정, 홈 화면 진입**
- `DriverHomeScreen.tsx` 로드
- **빈틈 3:** `getMyAssignment(today)` 호출 → 학원 관리자가 `VehicleAssignment`의 `driver_id`를 새 기사로 변경하지 않았으면 **"배정된 차량이 없습니다"** 빈 화면만 표시. 대차 기사는 무엇을 해야 하는지 전혀 알 수 없음
- 배정이 완료된 경우: 차량 번호판, 정원, 운영사, 안전도우미 이름 표시 → **이 정보 자체는 명확**
- 탑승/총 학생/완료 통계 카드 표시 → 첫 화면에서 "오늘 몇 명을 태워야 하는지" 즉시 파악 가능 (양호)

**07:57 — 루트(경로) 확인 시도**
- 하단 탭 "경로" 탭 클릭 → `DriverRouteScreen.tsx`
- 학생 목록이 순번별 카드로 표시: 학생 이름, 학원명, 픽업 시간
- AI 최적화 노선이 있으면 "AI 최적화 노선 v1 · 3.2km" 배너 표시
- **양호한 점:** 정류장 순서, 학생 이름, 픽업 시간이 한눈에 보임
- **빈틈 4:** 각 정류장의 **주소가 표시되지 않음**. `DriverDailyScheduleResponse` 스키마에 `pickup_latitude`, `pickup_longitude`만 있고 `pickup_address` 필드가 없음. 대차 기사는 "박유진 - 해오름학원 - 08:30"만 보이는데, **이 아이를 어디서 태워야 하는지 알 수 없음**
- **빈틈 5:** 각 카드에 **네비게이션(길찾기) 연동 버튼이 없음**. 지도 탭에 정류장 마커는 표시되지만, 개별 정류장에서 "카카오내비로 길안내" 같은 딥링크 기능이 없음. 초보 기사가 지도만 보고 길을 찾기는 매우 어려움

**08:00 — 지도 확인**
- "지도" 탭 → `DriverMapScreen.tsx`
- KakaoMap WebView에 정류장 마커와 현재 위치 표시
- GPS 활성 상태 인디케이터 표시
- **양호한 점:** 전체 루트를 지도에서 조감할 수 있음
- **빈틈 6:** 마커만 보이고 **턴바이턴 내비게이션이 아님**. 대차 기사에게는 "카카오내비/T맵 경유지 내비" 수준의 안내가 필요하나, 현재는 정적 마커 지도만 제공

**08:10 — 첫 정류장 도착, 학생 탑승 확인 시도**
- 정류장에 도착. 아이가 기다리고 있음
- **빈틈 7:** 학생 얼굴을 모름. `DriverDailyScheduleResponse`에 `student_name`만 있고 **프로필 사진(`profile_photo_url`)이 포함되지 않음**. DB의 `Student` 모델에는 `profile_photo_url` 필드가 있지만, 기사용 스케줄 API 응답에는 전달되지 않음. SRS 7.2에서 명시한 "프로필 사진 리스트 팝업"이 구현되어 있지 않음
- 기사가 이름을 부르고 아이가 응답하면 "탑승" 버튼을 눌러 확인 → **수동 확인만 가능**
- **빈틈 8:** SRS 6.2에서 요구한 **안면 인식 시스템(스마트 미러/태블릿)**이 구현되어 있지 않음 (Edge AI stub). 대차 기사처럼 학생 얼굴을 모르는 경우, 본인 확인 수단이 "이름 부르기"밖에 없음

**08:15 — 학부모가 "오늘 기사가 바뀌었다"는 사실을 아는가?**
- **빈틈 9:** 기사 변경 알림 시스템이 존재하지 않음. `notification/service.py`에는 탑승/하차/스케줄취소 알림만 구현. "기사 변경", "차량 변경" 이벤트에 대한 알림이 없음. 학부모는 모르는 기사가 아이를 태우러 왔을 때 불안감을 느낄 수밖에 없음
- VehicleAssignment 모델에 `driver_id` 변경 이력이나 변경 알림 트리거가 없음

**08:25 — 긴급 상황 발생 (가정: 아이가 차에서 넘어짐)**
- **빈틈 10:** 기사 앱에 **긴급 상황 대응 버튼(SOS/비상 호출)이 없음**. 4개 탭(홈/경로/지도/프로필)만 있고 비상 기능 없음
- **빈틈 11:** SRS 7.2에서 요구한 **"비상 오버라이드 알람"**(엣지 AI 이상 징후 감지 시 적색 섬광 + 경고음)이 구현되지 않음 (Edge AI는 stub)
- **빈틈 12:** 기사에게 **긴급 상황 대응 매뉴얼/가이드**가 앱 내에 없음. 대차 기사는 누구에게 연락해야 하는지, 어떤 절차를 밟아야 하는지 전혀 알 수 없음

**08:30 — Edge AI 기기 조작**
- **빈틈 13:** Edge AI 기기(NVIDIA Jetson, CCTV, 스마트 미러)가 차량에 설치되어 있다고 가정해도, **기기 조작 가이드가 앱 내에 없음**. edge_gateway 모듈은 `# Stub — Edge AI integration in M6`로 미구현 상태
- 대차 기사가 별도 교육 없이 AI 기기를 이해하고 조작하는 것은 불가능

## P22. 분리불안이 심한 저학년 아동의 학부모 — 한소희 (35세)

### 상황 배경

초등 1학년 딸 유진(7세)이 엄마와 떨어지면 울고 차에 타기 싫어함.
첫 셔틀 이용일. 아침부터 유진이가 울면서 안 가려고 함.

### 시간순 시뮬레이션

**14:00 — 사전 준비 (전날)**
- 한소희가 앱에서 내일 스케줄 확인 → `ParentHomeScreen.tsx`에서 자녀 스케줄 카드 확인
- 유진이의 스케줄: "15:00 픽업 예정"
- **빈틈 1:** 아이의 **특이사항(분리불안, 알레르기 등)을 등록할 수 있는 필드가 없음**. `Student` 모델에는 `name`, `date_of_birth`, `grade`, `profile_photo_url`만 있음. 기사나 안전도우미에게 "이 아이는 분리불안이 있으니 특별히 주의해 주세요"를 전달할 시스템적 방법이 없음
- **빈틈 2:** 첫 이용일인데 **사전 안내/준비 체크리스트가 없음**. "첫 탑승 전 아이에게 이렇게 설명해 주세요", "기사님 사진을 미리 보여주세요" 같은 안내 콘텐츠 부재

**14:50 — 픽업 10분 전, 유진이가 울기 시작**
- 한소희가 앱에서 차량 위치 확인 → `MapScreen.tsx` (학부모용)
- 실시간 추적 연결 상태 표시, "N정거장 전" ETA 표시
- **양호한 점:** 차량이 몇 정거장 전인지 확인 가능
- **빈틈 3:** "아이가 울고 있어 준비가 안 됐다"를 **기사에게 실시간으로 알릴 방법이 없음**. 학부모 → 기사 간 인앱 메시지/채팅 기능 없음. 전화번호도 앱에 표시되지 않음

**15:00 — 기사 도착, 유진이가 차에 타기를 거부**
- 기사가 정류장에 도착. 유진이가 울면서 안 타겠다고 함
- 기사가 억지로 태울 수는 없음 (아동 인권 문제)
- **빈틈 4:** 이 상황에서 기사가 따라야 할 **프로토콜이 없음**. "3분 대기 후 학부모 연락 → 5분 후 스킵" 같은 표준 절차 부재
- **빈틈 5:** 기사가 "이 정류장 스킵"을 앱에서 처리할 방법이 제한적. `markBoarded` / `markAlighted`만 있고, **"탑승 거부/불가" 상태 처리가 없음**. 현재 status는 `scheduled → boarded → completed` 또는 `cancelled`만 가능. "아이가 탑승을 거부함"이라는 상태가 없어서, 기사는 그냥 지나가야 하는데 스케줄은 "예정" 상태로 남음
- **빈틈 6:** 학부모가 앱에서 **당일 취소를 하려면** `ScheduleScreen.tsx`에서 "탑승 취소" 버튼을 눌러야 함. 취소 기능은 구현되어 있음 (`cancelSchedule` API). 그러나 **"아이가 차에서 거부하고 있는 바로 그 순간"에 학부모가 앱을 열어서 취소 버튼을 찾는 것은 비현실적**. 기사가 "탑승 불가" 버튼을 눌러서 학부모에게 자동 알림이 가는 플로우가 필요

**15:05 — 결국 "오늘은 안 보내겠다" 결정**
- 한소희가 앱에서 당일 취소 시도
- `ScheduleScreen.tsx` → 스케줄 카드 → "탑승 취소" 버튼 → 확인 다이얼로그 → 취소 완료
- `cancel_daily_schedule()` 호출 → status를 "cancelled"로 변경
- **양호한 점:** 당일 취소 프로세스 자체는 구현되어 있음
- **빈틈 7:** 취소 시 **취소 사유를 입력할 수 있으나 (`ScheduleCancelRequest.reason`)**, UI에서는 사유 입력 필드가 없음. `ScheduleScreen.tsx`의 `handleCancel`에서 `cancelSchedule(item.id)`만 호출하고, 사유는 전달하지 않음
- **빈틈 8:** 취소 알림이 **기사에게 실시간으로 전달되는지 불명확**. `send_schedule_cancelled_notification`은 학부모에게만 알림을 보냄. 기사에게 "유진이 픽업 취소됨, 해당 정류장 스킵하세요"라는 알림이 가지 않음. 기사는 경로 화면을 수동 새로고침해야 취소 상태를 알 수 있음

**15:10 — (가정) 유진이가 결국 차에 탔지만 차 안에서 계속 울고 있음**
- AI 이상행동 감지가 "우는 아이"를 어떻게 처리하는가?
- **빈틈 9:** Edge AI 모듈이 stub 상태. SRS 6.1에서 명시한 "이상 행동 감지"에 "울음"이 포함되는지도 명확하지 않음. SRS는 "돌아다니거나, 넘어짐, 폭력(싸움)"을 예시로 들고 있어, **분리불안으로 인한 울음은 "이상 행동"으로 분류되지 않을 가능성** 높음
- **빈틈 10:** 기사가 "아이가 울고 있어요"를 학부모에게 알릴 방법이 없음. 인앱 메시지/상태 공유 기능 없음. 기사가 직접 전화하려 해도 학부모 전화번호가 기사 앱에 표시되지 않음 (개인정보 보호 정책상 의도적일 수 있으나, 긴급 상황에서는 필요)

**15:20 — 유진이가 차에서 내리지 않으려 함 (목적지 도착 후)**
- 기사가 "하차" 버튼을 눌러야 하는데, 아이가 내리지 않으려 함

## 교차 분석: 두 페르소나가 동시에 발생하는 경우

**최악의 시나리오: 대차 기사 + 분리불안 아동**

최동훈(대차 기사)이 유진이(분리불안 아동)를 픽업하러 가는 상황:
- 유진이는 낯선 기사를 보고 더욱 공포에 질림
- 한소희(학부모)는 기사가 바뀐 것을 모름 (알림 없음)
- 최동훈은 유진이의 분리불안 특이사항을 모름 (정보 없음)
- 최동훈은 유진이 얼굴을 모름 (사진 없음)
- 최동훈은 긴급 상황 시 누구에게 연락해야 하는지 모름 (가이드 없음)
- 유진이가 울며 거부 → 최동훈이 어떻게 해야 할지 모름 (프로토콜 없음)

**이 시나리오에서 서비스가 제공하는 것:** 사실상 없음.
**이 시나리오에서 서비스에 필요한 것:** 거의 모든 것.
