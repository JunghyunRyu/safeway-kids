### CS담당A (이현지) 코멘트

> 학부모 문의의 80% 이상이 **"알림이 안 와요", "지도에 안 보여요", "기사가 누구예요"** 세 가지로 압축됩니다. 이 세 가지는 출시 전에 반드시 해결되거나, 최소한 CS에서 대응 가능한 매뉴얼이 있어야 합니다. 특히 "기사 정보 미노출"은 아이 안전과 직결되므로 학부모 불안 → 민원 폭주 → 서비스 신뢰 하락으로 이어질 수 있습니다.

### CS담당B (조영민) 코멘트

> 기사 측 문의는 **네비 연동 부재**가 가장 클 것입니다. 기사가 길을 못 찾으면 → 지연 → 학부모 문의 폭주로 연쇄됩니다. 학원 측은 **초기 세팅 병목**이 심각합니다. 학원 등록부터 학생 입력, 스케줄 설정까지 가이드가 없으면 CS로 전화가 쏟아질 것이고, 이건 전화 한 통에 30분 이상 걸리는 문의입니다.

## Part 2: 긴급 대응 매뉴얼 필요 항목

### 긴급도 1: 즉각 대응 필수 (골든타임 5분 이내)

| # | 시나리오 | 현재 시스템 지원 | CS 대응 매뉴얼 필요 사항 |
|---|----------|-----------------|------------------------|
| 1 | **학생 미탑승** — 정류장에 학생이 안 나옴 | DB에 `no_show` 상태 있으나, 기사 앱에 처리 UI 없음. 학부모/학원 자동 알림 없음 | 기사 → CS 전화 → CS가 학부모 + 학원에 유선 확인 → 스케줄 수동 상태 변경 절차 필요 |
| 2 | **차량 사고/고장** | SOS/긴급 버튼 전무. 긴급 알림 함수(`send_critical_alert_sms`)는 있으나 호출하는 UI/트리거 없음 | 기사 → 119/112 직접 신고 → CS에 전화 → CS가 학부모 전원에게 유선 연락 + 대체 차량 수배 절차 |
| 3 | **학생 하차 후 미도착** — 하차 완료 알림 갔으나 학원에 학생 안 옴 | 하차~학원 입구 사이 추적 불가. 도착 확인 시스템 없음 | 학부모/학원 → CS → 기사 + 안전도우미 유선 확인 → 현장 확인 → 상황에 따라 112 신고 |
| 4 | **차량 경로 이탈** — 정상 노선 벗어남 | 경로 이탈 감지 로직 없음. GPS 위치는 수집되나 이상 판단 없음 | 학부모가 지도에서 발견 → CS 전화 → CS가 기사 유선 확인 → 관제센터(플랫폼 관리자)에서 GPS 추적 |
| 5 | **안전도우미 미배정/불참** | EscortShift에 `no_show` 상태 있으나 자동 알림 없음 | CS가 아침 출근 시 배정 현황 수동 확인 → 미출근 시 대체 도우미 수배 |

### 긴급도 2: 30분 이내 대응

| # | 시나리오 | 현재 시스템 지원 | CS 대응 매뉴얼 필요 사항 |
|---|----------|-----------------|------------------------|
| 6 | **셔틀 15분 이상 지연** | 지연 감지/자동 알림 없음 | CS가 관제센터에서 GPS로 확인 → 기사 유선 확인 → 해당 노선 학부모 전원 SMS 발송 |
| 7 | **앱 장애 (서버 다운)** | 모니터링 시스템 미확인 | CS에서 장애 인지 → 개발팀 에스컬레이션 → 학부모/학원에 SMS 일괄 공지 |
| 8 | **결제 오류 / 이중 결제** | 결제 시뮬레이션 상태이므로 실제 발생 가능성은 PG 연동 후 | 학부모 → CS → 결제 로그 확인 → 환불 처리 (수동) |
| 9 | **기사 교체 (당일)** | 기사-차량 배정 UI 없음 | 학원/운수업체 → CS → 학부모 전원 SMS 안내 (새 기사 이름/연락처) |

### 긴급도 3: 당일 내 대응

| # | 시나리오 | 현재 시스템 지원 | CS 대응 매뉴얼 필요 사항 |
|---|----------|-----------------|------------------------|
| 10 | **학부모 회원 탈퇴 요청** | 사용자 비활성화는 관리자 대시보드에서 가능 | 동의 철회 → 스케줄 자동 비활성화 로직 있음(`withdraw_consent`) → CS가 대시보드에서 처리 |
| 11 | **청구 이의 제기** | 청구서 조회/수정은 대시보드에서 가능 | CS가 탑승 이력 조회 → 청구서 수정/재발행 |
| 12 | **개인정보 열람/삭제 요청** | 감사 로그에서 사용자 행동 추적 가능. 삭제는 soft delete | 개인정보보호법 대응 매뉴얼 필요 (30일 내 처리 의무) |

### CS담당B (조영민) 코멘트

> **가장 위험한 시나리오는 "학생 미탑승"과 "하차 후 미도착"입니다.** 이 두 가지는 시스템이 자동으로 감지하고 알림을 보내야 하는데, 현재는 기사가 수동으로 CS에 전화하는 것 외에 방법이 없습니다. 출시 전에 최소한 미탑승 처리 버튼과 자동 알림은 반드시 구현되어야 합니다. 그렇지 않으면 CS가 모든 긴급 상황의 허브가 되어 과부하가 걸립니다.

## Part 3: CS 도구 부족 목록

CS 담당자가 관리자 대시보드(`web/src/pages/platform/`)를 사용하여 고객 문의에 대응할 때 부족한 기능을 정리합니다.

### 있는 것 (현재 사용 가능)

| 기능 | 위치 | CS 활용도 |
|------|------|-----------|
| 사용자 CRUD (조회/등록/수정/삭제) | PlatformUsersPage | 높음 — 계정 문제 해결 |
| 사용자 역할 필터 (학부모/기사/관리자) | PlatformUsersPage | 높음 — 특정 역할 검색 |
| 학원 관리 (등록/수정) | PlatformAcademiesPage | 중 — 학원 정보 확인 |
| 차량 관리 (등록/수정) | PlatformVehiclesPage | 중 — 차량 정보 확인 |
| 청구서 관리 | PlatformBillingPage | 중 — 결제 문의 대응 |
| 감사 로그 (필터/페이지네이션/엑셀 내보내기) | PlatformAuditLogPage | 높음 — 이슈 추적 |
| 컴플라이언스 문서 관리 | PlatformCompliancePage | 낮음 — CS 직접 사용 드뭄 |
| 관제센터 (GPS 지도) | PlatformMapPage | 높음 — 위치 문의 대응 |
| 엑셀 내보내기 (사용자, 감사로그) | 각 페이지 ExportButton | 중 — 리포트 작성 |

### 없는 것 (CS 업무에 필수이나 미구현)

| # | 필요 기능 | 필요 이유 | 심각도 |
|---|----------|-----------|--------|
| 1 | **CS 티켓/문의 접수 시스템** | 학부모/기사/학원의 문의를 접수, 추적, 해결 상태 관리. 현재는 전화/카톡으로만 처리 가능 | 높음 |
| 2 | **학생 탑승 현황 대시보드** | "오늘 탑승 완료/미탑승/진행 중" 실시간 모니터링. 학부모가 "아이 탔나요?" 문의 시 즉시 확인 | 높음 |
| 3 | **알림 발송 이력 조회** | "알림 안 왔어요" 문의 시 해당 학생의 FCM/SMS 발송 성공/실패 이력 확인 | 높음 |
| 4 | **수동 알림 발송 기능** | 긴급 상황 시 특정 학부모/학원/기사에게 즉시 SMS/푸시 발송 | 높음 |
| 5 | **학생 상세 조회 (통합 뷰)** | 학생 이름으로 검색 → 보호자 정보, 등록 학원, 배정 차량, 오늘 스케줄, 탑승 이력, 청구서를 한 화면에서 확인 | 높음 |
| 6 | **기사-차량 배정 조회/변경** | "오늘 기사가 누구예요?" 문의 대응, 당일 기사 교체 처리 | 중 |
| 7 | **스케줄 수동 상태 변경** | 미탑승/취소/완료 등 스케줄 상태를 CS에서 직접 수정 | 중 |
| 8 | **FAQ/공지사항 관리** | 자주 묻는 질문 관리, 앱 내 공지 발송 | 중 |
| 9 | **학부모 연락처 빠른 검색** | 학생 이름으로 → 학부모 전화번호 즉시 조회 (긴급 연락용) | 높음 |
| 10 | **운행 지연 모니터링 알림** | 관제센터에서 예정 시간 대비 N분 지연 시 CS에 자동 알림 | 높음 |
| 11 | **일일/주간 CS 리포트** | 문의 유형별 건수, 평균 처리 시간, 미해결 건수 통계 | 낮음 |
| 12 | **메모/내부 코멘트** | 사용자/학원/차량에 CS 내부 메모 추가 (예: "이 학부모는 매주 금요일 취소 요청함") | 낮음 |

### CS담당A (이현지) 코멘트

> **가장 시급한 것은 "학생 통합 조회"입니다.** 학부모가 전화하면 "아이 이름이 김서준이에요"부터 시작하는데, 현재 대시보드에서 학생 이름으로 검색해서 → 보호자가 누구인지 → 오늘 스케줄이 뭔지 → 탑승했는지 → 어떤 차량인지를 한 번에 볼 수 있는 화면이 없습니다. 사용자 관리 페이지에서 학부모를 찾고, 학생 페이지에서 학생을 찾고, 스케줄 페이지에서 오늘 스케줄을 찾아야 합니다. 전화 한 통에 3~5분이 걸리면 콜센터는 마비됩니다.
>
> **두 번째로 시급한 것은 "알림 발송 이력"입니다.** "알림 안 왔어요"는 가장 흔한 문의인데, 현재 백엔드 로그에서만 확인 가능합니다. CS 담당자가 서버 로그를 볼 수는 없으니, 대시보드에 알림 이력 조회 기능이 있어야 합니다.

## Part 4: FAQ 초안

서비스 출시 시 앱 내 또는 웹사이트에 게시할 FAQ 항목입니다.

### 학부모 FAQ

**[가입/로그인]**

1. **Q: 앱은 어디서 다운로드하나요?**
   A: App Store(iPhone) 또는 Google Play(Android)에서 "세이프웨이키즈"를 검색하여 설치해주세요.

2. **Q: 가입은 어떻게 하나요?**
   A: 앱 설치 후 전화번호를 입력하면 인증번호(OTP)가 SMS로 발송됩니다. 인증번호를 입력하고 이름을 등록하면 가입이 완료됩니다.

3. **Q: 인증번호(OTP)가 안 와요.**
   A: 1) 전화번호를 정확히 입력했는지 확인해주세요. 2) 스팸 차단 앱이 SMS를 차단하고 있지 않은지 확인해주세요. 3) 3분 후 재시도해주세요. 계속 안 될 경우 고객센터(1588-XXXX)로 연락해주세요.

4. **Q: 배우자/조부모도 알림을 받을 수 있나요?**
   A: 현재는 주 보호자 1명의 기기로 알림이 발송됩니다. 보조 보호자 기능은 추후 업데이트 예정입니다. 당분간은 주 보호자가 가족에게 알림을 전달해주세요.

**[셔틀 이용]**

5. **Q: 셔틀은 몇 시에 오나요?**
   A: 앱 → 스케줄 탭에서 오늘 픽업 예정 시간을 확인할 수 있습니다. 실제 도착 시간은 교통 상황에 따라 다소 변동될 수 있습니다.

6. **Q: 셔틀이 늦어요. 어디쯤 있나요?**
   A: 앱 → 실시간 위치 탭에서 셔틀 버스의 현재 위치를 확인할 수 있습니다. 15분 이상 지연될 경우 고객센터로 연락해주세요.

7. **Q: 내일은 아이가 안 타는데 셔틀 취소는 어떻게 하나요?**
   A: 앱 → 스케줄 탭에서 해당 날짜의 스케줄을 선택하고 "취소" 버튼을 눌러주세요. 당일 취소는 픽업 시간 1시간 전까지 가능합니다.

8. **Q: 보충수업 때문에 하루만 추가로 태우고 싶어요.**
   A: 현재 앱에서 직접 추가는 불가합니다. 학원에 연락하시면 관리자가 스케줄을 추가해드립니다.

**[안전]**

9. **Q: 오늘 우리 아이를 태우는 기사님은 누구인가요?**
   A: 현재 앱에서 기사 정보 확인은 준비 중입니다. 학원에 문의하시면 담당 기사 정보를 안내받으실 수 있습니다.

10. **Q: 아이가 하차했다는 알림이 왔는데 학원에 도착했는지 어떻게 알 수 있나요?**
    A: 하차 알림은 셔틀에서 내린 시점에 발송됩니다. 학원 도착 확인은 학원 자체 출석 시스템으로 확인해주세요. 통합 도착 확인 기능은 추후 업데이트 예정입니다.

11. **Q: 긴급 상황 시 어디로 연락하나요?**
    A: 즉각적인 위험: 112(경찰) 또는 119(소방/구급)에 직접 신고해주세요. 서비스 관련 긴급 문의: 고객센터 1588-XXXX (운영 시간: 평일 07:00~20:00).

**[결제]**

12. **Q: 요금은 어떻게 계산되나요?**
    A: 탑승 건당 요금이 청구됩니다. 요금제는 학원별로 설정되며, 월간 상한이 있어 상한을 초과하는 금액은 청구되지 않습니다. 상세 요금은 학원에 문의해주세요.

13. **Q: 청구서는 언제 오나요?**
    A: 매월 초 전달 이용분에 대한 청구서가 앱으로 발송됩니다. 앱 → 청구서 탭에서 확인할 수 있습니다.

14. **Q: 자동결제 설정이 되나요?**
    A: 자동결제 기능은 준비 중입니다. 현재는 청구서 발송 후 앱에서 수동으로 결제해주세요.

## Part 6: 종합 의견

### CS담당A (이현지) 종합 의견

> 세이프웨이키즈는 백엔드와 관리 도구의 기본기가 잘 갖춰져 있습니다. 사용자 관리, 감사 로그, 컴플라이언스 문서 관리 등 운영에 필요한 기반은 마련되어 있어요. 하지만 **"고객 접점"에서의 완성도가 부족합니다.**
>
> 학부모가 가장 자주 묻는 3가지 — "알림이 안 와요", "위치가 안 보여요", "기사가 누구예요" — 에 대해 시스템이 스스로 답을 줄 수 없는 상태입니다. 이 3가지가 해결되지 않으면 출시 첫 주에 CS 전화가 폭주하고, 학부모 이탈이 시작됩니다.
>
> **CS 관점에서 출시 전 반드시 필요한 것:**
> 1. 기사/차량 정보 학부모 앱 노출
> 2. 알림 발송 실패 시 자동 재시도 + CS 대시보드에서 알림 이력 조회
> 3. 학생 통합 조회 화면 (이름 → 보호자, 학원, 스케줄, 탑승 상태 한 번에)
> 4. FAQ + 앱 내 도움말
>
> **CS 인원 최소 2명, 운영 시간 07:00~20:00으로 시작하되, 출시 첫 2주는 3명이 안전합니다.**

### CS담당B (조영민) 종합 의견

> 현장 운영 관점에서 가장 우려되는 것은 **긴급 상황 대응 체계의 부재**입니다. "세이프웨이"라는 이름에 걸맞게 안전이 최우선이어야 하는데, 현재 시스템에는:
>
> - SOS/긴급 버튼이 없습니다 (학부모, 기사, 학생 앱 전부)
> - 미탑승 자동 알림이 없습니다
> - 운행 지연 감지가 없습니다
> - 차량 내 잔류 학생 확인 절차가 없습니다
> - 하차 후 도착 확인이 없습니다
>
> 이런 상태에서 서비스를 출시하면, 사고 발생 시 "시스템이 알고 있었는데 알리지 않았다"는 비판을 받을 수 있습니다. 최소한 **SOS 버튼 + 미탑승 처리 + 지연 자동 알림**은 출시 전에 구현되어야 합니다.
>
> **기사 측에서는 네비 연동 없이는 서비스 자체가 불가능합니다.** 기사가 길을 못 찾으면 → 지연 → 학부모 불만 → CS 폭주 → 서비스 신뢰 하락의 악순환이 시작됩니다. 정류장 터치 → 카카오네비/T맵 실행, 이것 하나만 구현해도 기사 만족도와 운행 정시율이 크게 올라갑니다.
>
> **학원 온보딩도 CS에 큰 부담입니다.** 학원 관리자가 초기 설정(학생 등록, 스케줄 설정)을 스스로 할 수 없어서 모든 과정을 CS가 수동으로 도와줘야 합니다. 학원 1곳 온보딩에 최소 2~3시간이 소요될 것으로 예상합니다. 온보딩 마법사나 엑셀 업로드 개방으로 이 부담을 줄여야 합니다.
