### ITEM-P1-15: 운행 지연 자동 알림 — PASS (조건부)

**관련 법률:** 위치정보법 제15조 (개인위치정보 수집 동의), 제18조 (제3자 제공)

**적합 부분:**
- 지연 감지는 `DailyScheduleInstance`의 `pickup_time` 비교 로직이며, GPS 원본 데이터를 직접 학부모에게 전송하지 않음 — 위치정보 제3자 제공에 해당하지 않음
- FCM/SMS 알림 채널은 기존 인프라 활용이며, 지연 알림은 서비스 이용 목적 내 합리적 이용
- `delay_notified_at` 중복 방지 설계는 과잉 알림 방지 차원에서 적합

**조건부 사항:**

1. **알림 메시지에 "앱에서 차량 위치 확인" 유도 문구 — 위치정보 열람은 기동의된 범위 내에서만**
   - 기획서 알림 문구: "차량 위치를 앱에서 확인해주세요"
   - 위치정보 열람은 학부모의 `consent_scope.location_tracking = true` 동의가 전제
   - 동의 미부여 학부모에게는 해당 문구 제외 또는 "앱에서 운행 상태를 확인해주세요"로 변경 권고
   - **구현 시 동의 여부에 따른 알림 문구 분기 권고** (법적 의무는 아님, 혼란 방지)

2. **지연 감지 로직이 GPS 데이터에 의존하지 않음을 확인**
   - 현재 설계: `pickup_time + 10분 < now()` AND `status == 'scheduled'` — 시간 기반 판단
   - GPS 좌표를 활용한 "차량이 아직 먼 거리에 있음" 판단은 포함되지 않음 — 위치정보 추가 이용에 해당하지 않아 적합
   - 향후 GPS 기반 지연 감지 도입 시 위치정보 이용 목적 추가 고지 필요

### ITEM-P1-23: 학부모→기사/학원 메시지 기능 — NEEDS_AMENDMENT

**관련 법률:** 통신비밀보호법 제3조, 개인정보보호법 제17조 (제3자 제공), 정보통신망법 제50조 (영리목적 광고성 정보 전송 제한)

**적합 부분:**
- 200자 제한 단문 메시지: 비례성 원칙에 부합
- RBAC 기반 발송 권한 제한: 적합
- 학원 → 전체 학부모 공지: 서비스 운영 목적 내 합리적 이용

**수정 필요 사항:**

1. **메시지 보관 및 열람 관련 통신비밀보호법 준수**
   - 통신비밀보호법 제3조: "누구든지 ... 타인 간의 통신의 비밀을 침해하여서는 아니 된다"
   - 플랫폼 관리자/CS가 학부모-기사 간 메시지를 열람할 수 있는지에 대한 정책 명시 필요
   - **권고:**
     - CS/관제 목적 메시지 열람이 필요한 경우, 서비스 이용약관에 "분쟁 해결 및 안전 관리 목적으로 메시지 내용을 확인할 수 있습니다" 고지 필수
     - 메시지 보관 기간 명시 (권고: 6개월, 이후 자동 파기)
     - 기획서에 메시지 보관 정책 및 관리자 열람 범위 추가 필요

2. **기사 전화번호/개인정보 메시지 내 노출 방지**
   - 학부모가 기사에게 메시지를 보내는 구조에서, 수신자(기사) 정보가 학부모에게 어떻게 표시되는지 명확화 필요
   - 기사 실명 노출은 서비스 목적 내이나, **전화번호, 차량번호판 외 개인정보가 메시지 UI에서 노출되지 않도록** 설계 확인
   - 기사가 학부모에게 답신하는 기능이 있다면, 학부모 전화번호가 기사에게 노출되지 않아야 함 (현재 기획에 답신 기능 미포함 — 향후 추가 시 주의)

3. **학원 공지 발송 시 광고성 메시지 규제**
   - 학원 → 전체 학부모 공지가 **학원 자체 마케팅/광고** 목적으로 오용될 가능성
   - 정보통신망법 제50조: 영리목적 광고성 정보 전송 시 사전 동의 + 수신 거부 수단 제공 의무
   - **권고:**
     - 공지 내용에 "광고성 정보 전송 금지" 가이드라인 추가
     - 또는 공지 유형을 "운영 공지"로 한정하고, 마케팅/광고 전송은 별도 채널/동의 필요
     - 기획서에 공지 유형 제한 또는 광고성 메시지 차단 정책 추가

4. **메시지 데이터 개인정보보호법 적용**
   - `Message` 모델의 `content`는 개인정보가 포함될 수 있음 (아동 이름, 주소, 건강 정보 등)
   - 개인정보처리방침에 "서비스 내 메시지" 항목 추가 필요: 수집 항목, 보관 기간, 파기 방법
   - 기획서에 메시지 내 민감정보 필터링 또는 경고 문구 검토 권고 (P2 이상)

### ITEM-P1-24: CS 학생 통합 조회 화면 — NEEDS_AMENDMENT

**관련 법률:** 개인정보보호법 제17조 (제3자 제공), 제18조 (목적 외 이용), 제29조 (안전성 확보조치)

**적합 부분:**
- 플랫폼 관리자(RBAC) 전용 접근: 접근통제 원칙 충족
- CS 업무 목적의 통합 조회: 합목적적 이용 범위 내

**수정 필요 사항:**

1. **민감정보 표시 범위 제한**
   - 기획서: "학생 정보 + 보호자 정보 + 학원 정보 + 최근 7일 스케줄 + 최근 알림 이력" 한 화면 표시
   - CS 담당자에게 필요한 정보와 불필요한 정보 구분 필요:
     - **표시 적합:** 학생 이름, 학원 이름, 스케줄 상태, 알림 이력
     - **마스킹 필요:** 보호자 전화번호 (일부 마스킹: `010-****-5678`), 보호자 이메일
     - **표시 부적합 (별도 권한 필요):** 학생 생년월일 전체, 알레르기/특이사항(건강정보), 주소
   - 개인정보보호법 제16조(최소수집 원칙)의 취지: CS 목적에 필요한 최소 정보만 노출
   - **기획서에 CS 조회 화면의 정보 표시 범위 + 마스킹 기준 명시 필요**

2. **조회 이력 감사 로그**
   - CS 담당자의 학생 정보 조회는 개인정보 접근에 해당
   - 안전성 확보조치 기준 제8조: 개인정보 접속기록 최소 1년 보관 의무 (5만 명 이상 시 2년)
   - **기획서에 조회 이력 감사 로그 기록 명시 필요** (조회자, 조회 대상, 조회 시각, 조회 항목)

3. **검색 남용 방지**
   - 학생 이름/전화번호 대량 검색을 통한 개인정보 유출 가능성
   - 검색 결과 건수 제한 (예: 1회 검색 시 최대 10건) 또는 검색 빈도 제한 권고
   - 이상 검색 패턴 감지 시 보안 이벤트로 분류 (ITEM-REG-06 연동)

### ITEM-P1-25: 알림 발송 이력 조회 — PASS

CS 목적의 알림 이력 조회 기능. `NotificationLog` 모델의 설계가 적절하며, 플랫폼 관리자 전용 접근(RBAC)으로 접근통제 충족. 발송 이력 보관은 개인정보보호법 안전성 확보조치 기준에도 부합.

**권고 사항 (법적 의무는 아님):**
- `NotificationLog`에 수신자 전화번호(`recipient_phone`)가 평문으로 저장되는 경우, 마스킹 또는 해시 저장 검토 (이전 리뷰 P-03 관련)
- 알림 이력 보관 기간 명시 (권고: 1년, 이후 자동 파기)

### ITEM-P1-27: 운행 시작/종료 버튼 — PASS (조건부)

**관련 법률:** 위치정보법 제15조 (수집 동의), 제16조 (수집 기록), 근로기준법 제17조 (근로조건 명시)

**적합 부분:**
- GPS 추적의 시작/종료 시점을 기사가 명시적으로 제어: 위치정보 수집 범위의 명확화로 긍정적
- `RouteSession` 기록: 위치정보 수집 기록(위치정보법 제16조)의 기초 데이터로 활용 가능
- 운행 종료 시 GPS 중단: 불필요한 위치정보 수집 방지 — 최소수집 원칙에 부합

**조건부 사항:**

1. **운행 시작/종료와 위치정보 수집 기록 연동**
   - `RouteSession.started_at` = 위치정보 수집 시작 시점, `ended_at` = 수집 종료 시점
   - 위치정보법 제16조의 수집 기록으로 활용하려면, `RouteSession`에 수집 목적(`purpose = "safety_monitoring"`)도 기록 권고
   - ITEM-REG-07 (기사 위치정보 동의)과의 연동 확인 — 동의 미완료 시 운행 시작 차단이 기획에 포함되어 있어 적합

2. **운행 종료 후 GPS 데이터 잔류 방지**
   - 운행 종료 후 Redis의 실시간 위치 데이터가 즉시 삭제되는지 확인 필요
   - PostgreSQL `GpsHistory`의 보관 기간은 ITEM-REG-04 (위치정보 약관)에서 정의 (180일)

### ITEM-P1-28: 학생 특이사항/알레르기 기사 앱 표시 — NEEDS_AMENDMENT

**관련 법률:** 개인정보보호법 제23조 (민감정보 처리 제한), 동법 시행령 제18조

**적합 부분:**
- 아동 안전 확보 목적(차멀미 대응, 알레르기 응급 대처)의 정보 공유: 합목적적 이용
- 기사 앱에서의 표시(읽기 전용): 기사의 안전 의무 이행을 위한 최소 필요 정보

**수정 필요 사항:**

1. **건강정보는 개인정보보호법상 민감정보에 해당**
   - 개인정보보호법 제23조 제1항: "사상·신념, 노동조합·정당의 가입·탈퇴, 정치적 견해, **건강**, 성생활 등에 관한 정보 ... 정보주체에게 ... 별도의 동의를 받은 경우"에만 처리 가능
   - `allergies` (알레르기), `medical_notes` (의료 참고사항)는 **건강에 관한 민감정보**
   - 민감정보 처리를 위해서는 **별도의 동의**가 필요
   - **기획서에 다음 사항 추가 필요:**
     - 학부모(법정대리인)의 민감정보 수집 별도 동의 절차 — `GuardianConsent.consent_scope`에 `health_info_sharing: bool` 항목 추가
     - 동의 미부여 시 기사 앱에 알레르기/의료정보 미표시 (special_notes만 표시)
     - 동의서 문구: "아동의 건강 관련 정보(알레르기, 차멀미 등)를 운행 기사에게 제공하여 안전한 운행에 활용하는 것에 동의합니다"

2. **기사에 대한 민감정보 접근 범위 제한**
   - 기사에게 노출되는 정보: `special_notes` (일반 특이사항) + `allergies` (알레르기)
   - `medical_notes`는 의료 상세 정보로, 기사에게 노출할 필요가 없는 경우가 많음
   - **권고:** `medical_notes`는 기사 앱에 미표시 (학원 관리자/CS 전용으로 제한)
   - 기획서에 기사에게 노출되는 필드 범위를 명확히 한정 필요

3. **정보 최소화 및 표시 방식**
   - 알레르기 정보는 "경고 아이콘 + 키워드"로 최소 표시 (상세 의료 기록이 아닌 운행 주의사항 수준)
   - 기획서의 "빨간 경고 아이콘 + 텍스트" 방식은 적합하나, 표시 내용이 의료 진단명이 아닌 운행 주의사항임을 명확히 할 것

### ITEM-P1-32: 수동 SMS 발송 (CS 대시보드) — NEEDS_AMENDMENT

**관련 법률:** 개인정보보호법 제17조 (제3자 제공), 정보통신망법 제50조 (영리목적 광고 전송 제한), 전기통신사업법 제84조의2 (전화번호 이용)

**적합 부분:**
- 플랫폼 관리자 전용(RBAC): 접근통제 적합
- `NotificationLog` 연동: 발송 이력 기록으로 감사 추적 가능
- 발송 확인 팝업: 오발송 방지 적합

**수정 필요 사항:**

1. **수동 SMS의 발송 목적 제한 명시**
   - 기획서: "시스템 장애나 긴급 안내 시 수동으로 보낼 수 있어야 한다"
   - 그러나 구현상 자유 텍스트 입력(200자)이므로, 발송 목적 외 사용(개인적 연락, 광고 등) 가능성 존재
   - **기획서에 다음 추가 필요:**
     - 발송 목적 선택 드롭다운 추가: "긴급 안전 안내", "시스템 장애 공지", "운행 변경 안내", "기타 운영 안내"
     - 발송 사유가 `NotificationLog`에 함께 기록
     - 감사 로그에 발송자(관리자) ID + 발송 목적 + 수신자 정보 기록

2. **대량 발송 제한**
   - `recipient_ids: list[UUID]` — 수신자 목록에 제한이 없음
   - 전체 사용자 대상 대량 SMS 발송 시 비용 문제 + 스팸 우려
   - **권고:** 1회 발송 수신자 수 상한 설정 (예: 최대 50명, 전체 발송은 별도 승인 절차)
   - 대량 발송(50명 이상) 시 추가 관리자 승인 또는 이유 기록 강제

3. **수신자 전화번호 접근**
   - 수동 SMS 발송을 위해 백엔드에서 `User.phone` 조회 → SMS 발송
   - 관리자 UI에서 수신자 선택 시 전화번호가 화면에 노출되지 않도록 처리 (이름으로 검색/선택, 전화번호는 서버 측에서만 사용)
   - 기획서의 "수신자 검색 (이름/전화번호)" → **전화번호 검색은 서버 측 처리, UI에는 이름 + 역할만 표시** 권고
