## 1. 기술적 타당성 (20점 만점)

### 점수: 15 / 20

### 긍정 평가 (+15)

1. **모델 선택의 합리성:** YOLOv8n은 Ultralytics의 nano variant로서, CPU 추론에 최적화된 선택이다. COCO mAP 37.3 대비 파라미터 3.2M으로 경량-정확도 트레이드오프가 합리적이다 (Jocher et al., Ultralytics YOLOv8 Docs, 2023).

2. **ONNX Runtime 채택 결정:** PyTorch 직접 추론 대비 ONNX Runtime의 CPU 최적화(graph optimization, operator fusion)는 일반적으로 1.5~2x 속도 향상을 제공한다. 이 결정은 공학적으로 올바르다.

3. **시나리오별 모델 활성화 맵:** 동시 3모델 실행을 금지하고, 시나리오별로 필요한 모델만 활성화하는 설계는 i5 CPU의 컴퓨팅 버짓을 현실적으로 관리하는 접근이다.

4. **ArcFace ONNX 선택:** 히스토그램 매칭에서 ArcFace 512-dim embedding으로의 전환은 결정적으로 올바른 판단이다. ArcFace는 LFW 99.83%, CFP-FP 98.27%로 CPU 환경에서도 가장 신뢰할 수 있는 안면 인식 모델 중 하나이다 (Deng et al., "ArcFace: Additive Angular Margin Loss for Deep Face Recognition," CVPR 2019).

5. **적응형 프레임 스킵:** 추론 시간이 200ms를 초과할 때 2프레임을 스킵하는 적응형 전략은 실시간성과 정확도 사이의 균형을 잡는 실용적 접근이다.

### 감점 사유 (-5)

1. **추론 파이프라인의 end-to-end 지연시간 분석 부재 (-2):** 모델별 추론 시간은 명시되어 있으나, 전체 파이프라인(전처리 → 추론 → 후처리 → 렌더링 → WebSocket 전송)의 end-to-end 지연시간 분석이 없다. 개별 모델의 추론 시간이 100ms여도 전처리(리사이즈, 정규화), NMS, 바운딩 박스 렌더링, base64 인코딩, Socket.IO 전송까지 합산하면 실제 체감 지연은 150~250ms에 달할 수 있다.

2. **ONNX 모델 변환 검증 계획 부재 (-1.5):** YOLOv8n `.pt` → `.onnx` 변환 시 정확도 드롭(quantization error)에 대한 검증 계획이 없다. 특히 INT8 양자화를 적용할 경우 mAP 1~3% 하락이 일반적이며, 이에 대한 ablation 계획이 명시되어야 한다.

3. **MediaPipe PoseLandmarker의 한계 미언급 (-1.5):** MediaPipe Pose는 단일 인물 추적에 최적화되어 있으며, 다중 인물이 겹치는 환경(차량 내부 좌석)에서의 정확도 저하가 예상된다. BlazePose 모델의 occlusion 취약성에 대한 대응 전략이 없다 (Bazarevsky et al., "BlazePose: On-device Real-time Body Pose Tracking," CVPR Workshop 2020).

## 2. 수치적 근거 (20점 만점)

### 점수: 12 / 20

### 긍정 평가 (+12)

1. **추론 시간 목표의 현실성:** 시나리오별 예상 추론 시간(승하차 60~100ms, 이상행동 120~180ms, 잔류인원 80~130ms)은 YOLOv8n ONNX의 공식 벤치마크와 대체로 부합한다. Ultralytics 공식 문서에 따르면 YOLOv8n ONNX CPU 추론은 640x640 입력 기준 약 100~150ms이며, 416x416으로 축소 시 이보다 빠를 수 있다.

2. **ArcFace 인식률 근거:** LFW 99.83%는 Deng et al. (CVPR 2019) 논문에서 직접 보고된 수치로, 근거가 명확하다.

3. **FPS 목표 하향 조정의 합리성:** 원안의 5 FPS에서 "3 FPS 이상 (체감 5 FPS)"로 조정한 것은 R1 리뷰어의 지적을 반영한 현실적 수정이다.

### 감점 사유 (-8)

1. **95% 인식률의 테스트 조건 미정의 (-3):** "정면, 적정 조명 기준 95% 이상"이라는 목표치가 있으나, 구체적 테스트 프로토콜이 없다. 심사위원으로서 다음을 요구한다:
   - 테스트 피험자 수 (최소 N명)
   - 등록 이미지 수 대 테스트 이미지 수 비율
   - "적정 조명"의 정량적 정의 (lux 범위)
   - False Positive Rate (FAR)와 False Rejection Rate (FRR) 분리 보고
   - ArcFace의 LFW 99.83%는 통제된 벤치마크 데이터셋 결과이며, 실환경(웹캠, 다양한 조명, 어린이 얼굴)에서의 성능은 상당히 다를 수 있다. 이 격차에 대한 분석이 필수적이다.

2. **이상 행동 감지 "2초 이내" 근거 부재 (-2):** 이상 행동 감지 반응 시간 2초 이내라는 목표치의 공학적 근거가 없다. 추론이 프레임당 120~180ms이고, 이상 행동 판정에 temporal context (연속 프레임 분석)가 필요하다면, 몇 프레임을 축적해야 판정하는지, 그 결과 end-to-end 반응 시간이 어떻게 산출되는지 명시해야 한다.

3. **잔류 인원 감지 99% 정확도 근거 부재 (-2):** YOLOv8n의 COCO person class AP는 약 53% (AP@0.5:0.95)이며, AP@0.5 기준으로도 약 80% 수준이다. "99% 이상"이라는 목표치가 어떤 조건에서 달성 가능한지, recall과 precision을 분리하여 설명해야 한다. multi-frame 분석(5프레임 중 1프레임이라도 감지)으로 recall을 높이는 전략이라면, 그 수치적 모델링이 필요하다.

4. **벤치마크 재현 환경 미명시 (-1):** "Ultralytics 공식 벤치마크"를 근거로 인용하고 있으나, 해당 벤치마크의 CPU 사양(i7? Xeon? ARM?)이 i5와 동등한지 검증이 필요하다. 실제 i5-1235U와 i5-12400 사이에도 추론 시간 차이가 30~50% 발생할 수 있다.

## 3. 센서 시뮬레이션 정확성 (20점 만점)

### 점수: 14 / 20

### 긍정 평가 (+14)

1. **물리 기반 센서 모델링:** VLP-16 기반 LiDAR (360레이, 100m 범위)와 초음파 (12개 센서, 15도 빔, 4m 범위) 사양이 실제 상용 센서 사양과 부합한다. Velodyne VLP-16은 16채널, 360도 수평 FOV, 100m 범위 사양이 정확하다.

2. **가우시안 노이즈 모델 도입:** LiDAR σ=3cm, 초음파 σ=1cm, dropout 2%의 노이즈 모델은 시뮬레이션 사실성을 크게 높인다. 실제 VLP-16의 range accuracy ±3cm (typical)과 일치한다 (Velodyne VLP-16 Datasheet).

3. **센서 융합 가중 평균:** LiDAR 0.7 + 초음파 0.3의 규칙 기반 융합은 PoC 수준에서 적절하다. 칼만 필터나 파티클 필터 등 고급 융합은 PoC 범위를 초과한다.

4. **경고 거리의 물리적 타당성 수정:** R3 리뷰어의 지적을 반영하여 경고 거리를 센서 범위 내로 조정한 것(초음파 주의 3.5m / 경고 2m / 위험 0.8m)은 물리적으로 올바른 수정이다.

5. **어린이 객체 물리 파라미터:** 높이 90~130cm, 이동속도 0.5~1.5 m/s는 3~7세 어린이의 일반적 신체 조건 및 보행 속도와 일치한다.

### 감점 사유 (-6)

1. **LiDAR 반사 강도(intensity) 모델 부재 (-2):** 실제 LiDAR는 거리뿐 아니라 반사 강도(reflectivity)를 반환한다. 어린이의 의복 색상/재질에 따라 반사 강도가 달라지며, 이는 감지 신뢰도에 영향을 준다. 시뮬레이션에서 반사 강도 모델이 없으면, 실제 센서 전환 시 예상치 못한 감지 실패가 발생할 수 있다.

2. **초음파 멀티패스/크로스토크 모델 부재 (-1.5):** 12개 초음파 센서를 동시 운용할 때, 인접 센서 간 크로스토크(cross-talk)와 벽면/차체 반사에 의한 멀티패스(multipath) 현상은 실제 환경에서 주요 오차 원인이다. 이에 대한 시뮬레이션이 없으면 "시뮬레이션이 너무 이상적"이라는 비판을 받을 수 있다 (Borenstein & Koren, "Obstacle Avoidance with Ultrasonic Sensors," IEEE Journal of Robotics and Automation, 1988).

3. **날씨/환경 조건 변수 부재 (-1.5):** 비, 안개, 먼지 등 환경 조건이 LiDAR 성능에 미치는 영향에 대한 시뮬레이션 파라미터가 없다. PoC이므로 필수는 아니나, "향후 확장" 섹션에서라도 언급해야 기술적 깊이를 보여줄 수 있다. Heinzler et al. ("Weather Influence and Classification with Automotive Lidar Sensors," IEEE IV 2019)에 따르면 강우 시 LiDAR 감지 거리가 최대 40% 감소할 수 있다.

4. **센서 융합의 시간 동기화(temporal alignment) 미고려 (-1):** LiDAR(10~20Hz)와 초음파(25~40Hz)는 샘플링 주기가 다르다. 실제 융합에서는 시간 동기화가 핵심이며, 시뮬레이션에서도 이를 반영해야 한다. 현재 계획서에서는 두 센서의 데이터가 동시에 사용 가능한 것으로 암묵적으로 가정하고 있다.

## 4. 확장성 경로 (20점 만점)

### 점수: 11 / 20

### 긍정 평가 (+11)

1. **ONNX Runtime 기반 설계:** ONNX는 하드웨어 중립적 추론 포맷으로, CPU에서 개발 후 NVIDIA Jetson의 TensorRT EP(Execution Provider)로 전환 시 코드 변경이 최소화된다. 이 아키텍처 결정은 확장성을 고려한 것이다.

2. **모듈화된 시나리오 설계:** 4개 시나리오가 독립적으로 동작하며, 시나리오별 모델 활성화 맵으로 관리되는 구조는 실제 Jetson 배포 시에도 유효하다.

3. **Standalone/연동 모드 분리:** 백엔드 자동 감지로 독립 실행과 연동 모드를 모두 지원하는 설계는 현장 배포 시 네트워크 불안정 상황에 대비한 올바른 접근이다.

### 감점 사유 (-9)

1. **Jetson 마이그레이션 계획의 구체성 부족 (-3):** "ONNX Runtime → TensorRT"라는 경로만 암시되어 있을 뿐, 구체적 마이그레이션 계획이 없다:
   - Jetson Orin Nano의 GPU 메모리(8GB)에서 4개 모델 동시 로딩 가능 여부
   - TensorRT FP16/INT8 변환 시 정확도 검증 계획
   - CUDA 버전, JetPack 버전 호환성
   - CPU PoC → GPU 프로덕션 전환 시 배치 사이즈, 메모리 레이아웃 변경 사항

2. **실 센서 인터페이스 설계 부재 (-3):** 소프트웨어 시뮬레이션에서 실제 LiDAR(ROS2 point cloud topic), 초음파(GPIO/UART), CCTV(RTSP)로의 전환 시 필요한 인터페이스 추상화 계층(HAL)이 설계되어 있지 않다. 현재 JS 프론트엔드에서 시뮬레이션하는 센서 데이터를, 실제 센서 드라이버로 교체하려면 상당한 리아키텍처가 필요할 것으로 보인다.

3. **확장 시 네트워크 아키텍처 미설계 (-2):** 차량 1대에서 수십~수백 대로 확장 시, 각 차량의 Jetson에서 중앙 서버로의 데이터 전송 아키텍처(MQTT, gRPC 등), 대역폭 요구량, 지연시간 SLA 등이 전혀 논의되지 않았다.

4. **모델 업데이트/OTA 전략 부재 (-1):** 프로덕션 환경에서 AI 모델의 OTA(Over-the-Air) 업데이트, A/B 테스트, 롤백 전략이 없다. MLOps 관점의 확장성이 누락되어 있다.

## 5. 데모 기술 완성도 (20점 만점)

### 점수: 16 / 20

### 긍정 평가 (+16)

1. **스텝 위저드 패턴:** 시나리오 간 전환을 위저드 형태로 설계한 것은 데모의 서사적 흐름(narrative flow)을 보장하며, 비기술 심사위원도 따라가기 쉽다.

2. **프레젠터 가이드 UI:** 각 시나리오 설명 패널을 포함한 것은 발표자(대표)가 심사위원에게 기술을 설명하면서 동시에 시연할 수 있게 하는 실용적 설계이다.

3. **Fallback 전략:** 웹캠 실패 시 사전 녹화 영상 재생은 데모 안정성의 핵심이다. 규제 심사에서 기술적 장애로 시연이 중단되면 치명적이므로, 이 결정은 매우 중요하다.

4. **성능 대시보드:** CPU, 메모리, FPS, 추론 시간을 실시간으로 표시하는 것은 기술 심사위원에게 투명성을 보여주는 효과적 방법이다.

5. **모델 로딩 스플래시 화면:** warm-up 과정을 시각적으로 보여주는 것은 "대기 시간"을 "기술 시연"으로 전환하는 영리한 UX 판단이다.

6. **WebSocket 전환:** MJPEG에서 Socket.IO base64 전송으로 전환한 것은 양방향 통신과 이벤트 기반 아키텍처를 가능하게 하며, 기술적으로 올바른 진화이다.

### 감점 사유 (-4)

1. **실시간 추론 품질의 시각적 신뢰도 우려 (-2):** CPU 3 FPS에서의 바운딩 박스 오버레이는 시각적으로 "끊기는" 느낌을 줄 수 있다. 추론 결과를 최신 프레임에 보간(interpolation)하여 오버레이하는 전략이 명시되어 있지만, 구체적 구현 방법(칼만 필터 기반 예측? 단순 좌표 유지?)이 없다. 심사위원이 "느린 AI"라고 인식하면 감점 요인이 된다.

2. **데모 영상 녹화본 품질 관리 미계획 (-1):** Consensus Matrix에서 데모 영상 녹화본 제출 여부가 미결(Q10)이나, 실제 규제 심사에서는 녹화본을 보조 자료로 요구하는 경우가 많다. 고품질 녹화본(4K, 편집, 자막)의 제작 계획이 필요하다.

3. **동시 시각화 부하 미측정 (-1):** Canvas 2D 사각지대 렌더링 + WebSocket 영상 스트리밍 + 성능 대시보드 차트가 동시에 동작할 때의 브라우저 렌더링 성능(jank, dropped frames)에 대한 프로파일링 계획이 없다.

## 90점 이상 달성을 위해 반드시 보강해야 할 사항

### 보강 1: 정량적 성능 검증 프로토콜 수립 (+8점 가능)

**현재 문제:** 95% 인식률, 200ms 추론, 99% 잔류인원 감지 등 핵심 수치에 대한 측정 프로토콜이 부재하다.

**보강 내용:**
- **안면 인식 테스트 프로토콜:** 최소 10명 피험자 x 각 3장(정면, 좌15도, 우15도) 등록, 테스트 시 인당 10회 시도. FAR과 FRR을 분리 보고. 조명 조건을 200lux, 500lux, 1000lux로 구분.
- **이상 행동 감지 프로토콜:** "서기" 행동의 정의(hip-shoulder-knee 각도 threshold), 연속 N프레임 판정 기준, TPR/FPR 측정.
- **잔류 인원 감지 프로토콜:** 5프레임 multi-frame voting에서의 수학적 기대치 도출. 단일 프레임 recall이 r일 때, 5프레임 중 1회 이상 감지 확률 = 1-(1-r)^5 임을 명시.

**참고 논문:** Grother et al., "Face Recognition Vendor Test (FRVT)," NIST IR 8280, 2019 — 안면 인식 벤치마크 표준 프로토콜.

### 보강 2: End-to-End 파이프라인 지연시간 분석 (+5점 가능)

**현재 문제:** 모델별 추론 시간만 있고, 전처리·후처리·렌더링·전송의 총 지연시간이 분석되지 않았다.

**보강 내용:**
```
전체 지연시간 = 프레임 캡처(~10ms) + 리사이즈/정규화(~5ms) + ONNX 추론(80~180ms)
              + NMS/후처리(~10ms) + 오버레이 렌더링(~5ms) + base64 인코딩(~15ms)
              + Socket.IO 전송(~5ms) + 브라우저 디코딩/렌더링(~10ms)
              ≈ 140~240ms (end-to-end)
```
이런 형태의 타이밍 다이어그램을 제시하고, 실측 데이터로 검증하는 계획을 명시해야 한다.

**참고:** Redmon & Farhadi, "YOLOv3: An Incremental Improvement," arXiv:1804.02767, 2018 — 추론 파이프라인 벤치마크 방법론 참고.

### 보강 3: 센서 시뮬레이션 물리 모델 강화 (+4점 가능)

**현재 문제:** 가우시안 노이즈만 있고, 반사 강도, 크로스토크, 환경 조건 모델이 없다.

**보강 내용:**
- **LiDAR 반사 강도 모델:** 객체 재질별 반사율 (어린이 의복 0.3~0.7, 아스팔트 0.1~0.2, 금속 차체 0.8~0.95) 파라미터화. 거리에 따른 반사 강도 감쇠: I = I₀ × ρ / r² (역제곱 법칙).
- **초음파 크로스토크 모델:** 인접 센서 억제 시간(blanking time) 시뮬레이션. 일반적으로 50~100ms의 blanking time을 적용.
- **환경 파라미터(nice-to-have):** 강우/안개 시 감쇠 계수. Beer-Lambert 법칙 기반: I = I₀ × exp(-αd), α는 기상 조건별 감쇠 계수.

**참고 논문:**
- Hasirlioglu et al., "Modeling and Simulation of Rain for the Test of Automotive Sensor Systems," IEEE IV 2016
- Borenstein & Koren, "Obstacle Avoidance with Ultrasonic Sensors," IEEE JRA 1988

### 보강 4: Jetson 마이그레이션 로드맵 구체화 (+3점 가능)

**현재 문제:** "ONNX를 쓰니까 Jetson으로 옮기면 된다" 수준의 언급만 있다.

**보강 내용:**
- **Phase 1 (PoC, 현재):** i5 CPU + ONNX Runtime CPU EP → 기능 검증
- **Phase 2 (하드웨어 통합):** Jetson Orin Nano + TensorRT EP → 성능 검증
  - 예상 추론 시간: YOLOv8n TensorRT FP16 on Jetson Orin Nano ≈ 3~5ms/frame (Ultralytics Jetson benchmark)
  - JetPack 6.0, CUDA 12.2, TensorRT 8.6 호환성 명시
- **Phase 3 (센서 통합):** ROS2 드라이버 + Velodyne VLP-16 + HC-SR04 초음파
  - HAL(Hardware Abstraction Layer) 인터페이스 정의: `SensorDriver.read() -> SensorFrame`
  - RTSP 카메라 입력: GStreamer → OpenCV 파이프라인
- **Phase 4 (Fleet 확장):** MQTT 기반 차량-서버 통신, 차량당 bandwidth ≈ 50Kbps (이벤트 전용)

**참고:** NVIDIA, "Jetson Orin Nano Developer Kit — Performance Benchmarks," 2023

### 보강 5: 바운딩 박스 보간 및 시각적 품질 개선 (+2점 가능)

**현재 문제:** 3 FPS 추론으로 인한 시각적 끊김 우려.

**보강 내용:**
- **칼만 필터 기반 바운딩 박스 예측:** 추론 프레임 사이의 중간 프레임에서 이전 추론 결과를 칼만 필터로 보간하여 "체감 15 FPS" 이상의 부드러운 오버레이 달성.
- **참고:** Bewley et al., "Simple Online and Realtime Tracking," ICIP 2016 (SORT 알고리즘) — 칼만 필터 기반 객체 추적의 표준 접근.

## 기술 심사위원으로서 반드시 물을 질문 5개 + 기대 답변

### 질문 1: "95% 안면 인식률은 어떤 조건에서의 수치입니까? 벤치마크 데이터셋 결과와 실환경 결과를 구분해서 설명해 주세요."

**기대 답변:**
"ArcFace 모델의 LFW 벤치마크 결과는 99.83%입니다. 다만 이는 통제된 환경의 정적 이미지 데이터셋 결과이며, 실환경(웹캠, 다양한 조명, 어린이 얼굴)에서는 성능이 저하될 수 있습니다. 저희는 PoC에서 최소 10명의 피험자를 대상으로 200lux~1000lux 조명 환경에서 테스트하여, FAR < 1%, FRR < 5%를 목표로 합니다. 다중 각도 등록(정면+좌15도+우15도)과 CLAHE 전처리로 실환경 격차를 보완합니다. 정량적 테스트 결과를 별도 보고서로 제출하겠습니다."

### 질문 2: "소프트웨어 센서 시뮬레이션이 실제 LiDAR/초음파와 얼마나 다를 수 있다고 보십니까? Sim-to-Real gap을 어떻게 관리하실 계획입니까?"

**기대 답변:**
"PoC 시뮬레이션은 거리 측정의 가우시안 노이즈(LiDAR σ=3cm, 초음파 σ=1cm)와 랜덤 dropout(2%)을 포함하여 기본적인 센서 불확실성을 모델링합니다. 다만 실제 환경에서 발생하는 멀티패스 반사, 날씨 영향, 초음파 크로스토크는 현재 모델에 포함되지 않습니다. Phase 2에서 실 센서 데이터 수집 후, 시뮬레이션 파라미터를 실측 데이터 기반으로 보정(calibration)할 계획입니다. 현 단계에서는 '보수적 파라미터'(실제보다 더 많은 노이즈, 더 짧은 유효 범위)를 적용하여 과대 추정을 방지합니다."

### 질문 3: "i5 CPU에서 3 FPS라면, 실제 Jetson 배포 시에는 몇 FPS를 기대하며, 그 근거는 무엇입니까?"

**기대 답변:**
"Jetson Orin Nano에서 TensorRT FP16으로 YOLOv8n을 실행하면 약 200~300 FPS가 가능합니다 (NVIDIA 공식 벤치마크 기준). 멀티 모델 동시 실행 시에도 30 FPS 이상 달성이 가능합니다. 현재 PoC의 3 FPS는 CPU 제약에 의한 것이며, 동일 ONNX 모델을 TensorRT EP로 전환하는 것만으로 50~100배 성능 향상이 예상됩니다. 이는 Ultralytics의 Jetson 벤치마크 문서에서 확인 가능합니다."

### 질문 4: "이상 행동 감지에서 '일어서기'와 '넘어짐'을 어떻게 구분합니까? False positive 비율은 어떻게 관리합니까?"

**기대 답변:**
"MediaPipe PoseLandmarker의 33개 관절 좌표 중 hip, shoulder, knee, ankle 좌표를 활용합니다. '일어서기'는 hip-y 좌표가 기준선 대비 일정 threshold 이상 상승하고, knee 각도가 펴진 상태(>150도)가 N프레임 이상 지속될 때 판정합니다. '넘어짐'은 hip-y가 급격히 하강하고 shoulder-hip 벡터가 수평에 가까워질 때 판정합니다. False positive 감소를 위해 5프레임 temporal smoothing(majority voting)을 적용하며, PoC에서 실측한 TPR/FPR을 보고서에 포함할 예정입니다."

### 질문 5: "이 PoC에서 실제 프로덕션까지의 기술적 격차(gap)를 상위 3개만 말씀해 주세요. 그리고 각각의 해결 계획은?"

**기대 답변:**
"첫째, **하드웨어 통합**입니다. CPU 시뮬레이션에서 Jetson + 실 센서로 전환해야 합니다. ONNX 기반 설계로 AI 추론 전환은 저비용이나, 센서 드라이버 통합(ROS2)은 별도 개발이 필요합니다. 둘째, **대규모 운영**입니다. 차량 1대 PoC에서 수백 대 fleet으로 확장 시, 에지-클라우드 아키텍처 설계, OTA 모델 업데이트, 모니터링 시스템이 필요합니다. 셋째, **실환경 정확도 검증**입니다. 통제된 데모 환경과 달리 다양한 조명, 날씨, 도로 조건에서의 정확도를 대규모 필드 테스트로 검증해야 합니다. 각각에 대해 Phase 2(하드웨어 통합 3개월), Phase 3(파일럿 운영 6개월), Phase 4(상용화 12개월)로 로드맵을 수립했습니다."

## 종합 의견

### 현재 수준: 68/100 — 조건부 보완 필요

본 PoC 계획서는 **기술적 방향성은 올바르나, 규제 심사를 통과하기 위한 정량적 엄밀성이 부족**하다.

**강점:**
- AI 모델 선택(YOLOv8n, ArcFace ONNX)과 추론 파이프라인 설계가 공학적으로 합리적이다
- 센서 시뮬레이션의 물리 파라미터가 실제 센서 사양에 기반한다
- 데모 UX 설계(위저드, fallback, 프레젠터 가이드)가 심사 환경에 적합하다
- Consensus Matrix를 통한 4명의 전문가 리뷰와 합의 도출 과정이 체계적이다

**약점:**
- 핵심 성능 수치(95% 인식률, 200ms 추론, 99% 잔류 감지)에 대한 **측정 프로토콜과 검증 계획이 부재**하다. 수치를 제시하면서 측정 방법을 제시하지 않는 것은 기술 심사에서 신뢰도를 떨어뜨린다.
- **PoC에서 프로덕션으로의 마이그레이션 경로**가 모호하다. "ONNX를 쓰니까 쉽게 옮길 수 있다"는 수준의 설명으로는 기술 심사위원을 설득하기 어렵다.
- **센서 시뮬레이션이 이상적 환경만을 가정**하고 있어, 실제 운영 환경과의 격차(sim-to-real gap)에 대한 인식이 부족해 보인다.

**권고:**
위 5가지 보강 사항을 반영하면 현재 68점에서 **약 88~90점**까지 도달 가능하다. 특히 **보강 1(정량적 성능 검증 프로토콜)**과 **보강 4(Jetson 마이그레이션 로드맵)**는 기술 심사위원이 가장 집중적으로 심사하는 영역이므로 반드시 Tech Spec에 포함해야 한다. PoC의 기술적 방향은 옳으므로, 엄밀성과 구체성을 보강하면 심사 통과 가능성이 높다.
