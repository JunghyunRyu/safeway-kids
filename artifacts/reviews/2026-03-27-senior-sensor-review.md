## 1. 요구사항 재진술

도메인 4는 어린이 통학버스 주변의 **사각지대(blind spot)**를 감지하기 위한 LiDAR + 초음파 센서 시스템을 **소프트웨어로 시뮬레이션**하는 것이다. 실제 하드웨어(LiDAR 센서, 초음파 센서)는 사용하지 않으며, 규제 샌드박스 심사위원에게 다음을 입증해야 한다:

1. **센서 원리 이해:** LiDAR 레이 캐스팅(ray casting)과 초음파 콘(cone) 감지의 물리적 원리를 정확히 시뮬레이션
2. **센서 융합(Sensor Fusion):** 두 종류의 센서 데이터를 결합하여 단일 센서보다 높은 신뢰도의 객체 감지 의사결정
3. **단계별 경고 체계:** 어린이가 차량에 접근할 때 거리 기반 3단계 경고 (주의 5m / 경고 3m / 위험 1.5m)
4. **Bird's Eye View 시각화:** 차량 주변 360도 감지 범위를 조감도(top-down view)로 직관적으로 표현
5. **시나리오 재생:** 사전 정의된 시나리오를 안정적으로 자동 재생하여 데모 실패 리스크 최소화

핵심 목적은 "이 기술을 실제 차량에 탑재하면 어린이 사각지대 사고를 예방할 수 있다"는 것을 비기술 심사위원이 납득하도록 하는 것이다.

## 2. 누락된 요구사항

### 2.1 센서 사양 명세 부재 (Critical)

현재 요구사항에는 시뮬레이션할 센서의 **구체적 사양**이 전혀 없다. 규제 심사위원은 "어떤 센서를 사용할 것인가?"를 반드시 물을 것이다.

| 항목 | 현재 상태 | 필요한 명세 |
|------|----------|------------|
| LiDAR 모델 | 미명시 | Velodyne VLP-16급: 16채널, 360도, 100m 범위, ±15도 수직 FOV |
| LiDAR 각도 해상도 | 미명시 | 수평 0.1~0.4도, 수직 2.0도 |
| LiDAR 포인트 밀도 | 미명시 | ~300,000 points/sec (10Hz 회전 시) |
| 초음파 센서 모델 | 미명시 | HC-SR04급: 40kHz, 2~400cm 범위, 15도 빔 각도 |
| 초음파 센서 개수 | 미명시 | 최소 8개 (전방 4, 후방 4) 또는 12개 (측면 포함) |
| 센서 배치 위치 | 미명시 | 차량 바디 기준 정확한 좌표 필요 |
| 차량 치수 | 미명시 | 어린이 통학버스: 폭 ~2.0m, 길이 ~7.0m, 높이 ~2.8m |

### 2.2 어린이 객체 모델 부재 (Important)

감지 대상인 "어린이"의 물리적 특성이 정의되지 않았다:

- **신체 크기:** 3~7세 기준 높이 90~130cm, 폭 30~40cm, 깊이 20~25cm
- **레이더 단면적(RCS):** 어린이는 성인보다 작아 감지가 어려움
- **반사 특성:** 의복 재질에 따른 LiDAR 반사율 차이 (반사 조끼 vs 일반 의류)
- **이동 속도:** 걷기 0.5~1.5 m/s, 달리기 2~4 m/s

### 2.3 환경 조건 명세 부재 (Important)

실제 운용 환경에서의 센서 성능 저하 요인이 고려되지 않았다:

- **날씨:** 비, 눈, 안개 시 LiDAR 성능 저하 (최대 30% 범위 감소)
- **주변 장애물:** 주차된 차량, 기둥, 벽 등에 의한 가림(occlusion)
- **다중 객체:** 어린이 여러 명이 동시에 접근하는 경우
- **지면 반사:** LiDAR 지면 반사(ground return) 필터링

### 2.4 센서 노이즈 모델 부재 (Important)

실제 센서는 완벽하지 않다. 시뮬레이션의 공학적 신뢰성을 위해 노이즈 모델이 필수다:

- **LiDAR:** 거리 측정 노이즈 (±2~3cm @100m), 각도 지터(jitter), 누락 포인트(dropout)
- **초음파:** 측정 노이즈 (±1cm), 크로스토크(crosstalk), 다중 경로 반사(multipath)
- **위양성(False Positive):** 센서가 없는 객체를 감지하는 경우
- **위음성(False Negative):** 센서가 있는 객체를 놓치는 경우

### 2.5 응답 시간 세분화 부재 (Minor)

"500ms 이내"라는 전체 응답 시간 목표는 있으나, 파이프라인 단계별 시간 예산이 없다:

| 단계 | 권장 시간 예산 |
|------|--------------|
| 센서 데이터 수집 | ~50ms (시뮬레이션에서는 즉시) |
| 포인트 클라우드 처리 | ~100ms |
| 객체 감지/분류 | ~150ms |
| 센서 융합 결정 | ~50ms |
| UI 렌더링 | ~100ms |
| 여유분 | ~50ms |
| **합계** | **~500ms** |

## 3. 충돌/모순

### 3.1 감지 범위와 경고 거리의 불일치

- 초음파 센서(HC-SR04급) 최대 감지 거리: **4m** (실용 범위)
- 요구사항의 "주의" 경고 거리: **5m**
- **모순:** 초음파 센서 단독으로는 5m에서 감지할 수 없다. LiDAR는 가능하지만, 이 경우 5m 주의 경고는 LiDAR 전용이고 초음파는 3m 이하에서만 작동한다는 것을 명시해야 한다.

**해결 제안:** 센서별 유효 감지 범위를 구분하고, 경고 단계를 센서 가용성에 연동:
- 5m (LiDAR only): 주의 — "LiDAR 감지"
- 3m (LiDAR + 초음파): 경고 — "센서 융합 확인"
- 1.5m (LiDAR + 초음파 고신뢰): 위험 — "즉시 정지"

### 3.2 "360도 감지"와 "전후방 센서" 명세 충돌

- AC4에서 "360도 감지 범위 시각화"를 요구
- 그러나 도메인 4 테이블에서 "차량 전후방" LiDAR/초음파만 언급
- LiDAR가 360도 회전형이라면 360도 커버 가능하나, 초음파는 전후방에만 설치되므로 측면 사각지대가 존재

**해결 제안:** LiDAR는 360도 커버, 초음파는 전방 4 + 후방 4 + 측면 4 = 12개 배치로 360도 근접 감지 보완. Bird's Eye View에서 각 센서의 커버리지를 다른 색상으로 표시.

### 3.3 인터랙티브 vs 자동 재생

- Q3 (Open Question)에서 마우스 수동 이동 vs 자동 시나리오 재생을 미결로 남김
- AC4에서는 "시나리오 자동 재생 모드 지원"만 명시
- 규제 심사 상황에서는 **자동 재생이 1순위** (실패 리스크 최소화), 마우스 인터랙션은 보조 기능

**해결 제안:** 자동 재생을 기본 모드로, 마우스 인터랙션을 "전문가 모드"로 별도 제공. 데모 시에는 자동 재생만 사용.

## 4. 기술 리스크

### 4.1 "단순 애니메이션" 리스크 (R5, 매우 높음)

이것이 **도메인 4의 최대 리스크**다. 규제 심사위원이 "이건 그냥 움직이는 그림 아닌가?"라고 판단하면 PoC 전체의 신뢰도가 훼손된다.

**리스크 완화 전략:**

1. **물리 기반 레이 캐스팅(Ray Casting):** 단순 원형 범위 표시가 아니라, LiDAR에서 실제로 수백 개의 레이를 쏘아 장애물에 반사되는 것을 시각적으로 보여줘야 한다. 레이가 객체에 부딪히면 포인트가 생기고, 빈 공간에는 레이가 최대 거리까지 뻗는 것이 보여야 한다.

2. **센서 노이즈 시각화:** 완벽한 원형이 아니라 약간 흔들리는(jittering) 포인트, 가끔 빠지는(dropout) 포인트가 있어야 실제 센서처럼 보인다. 완벽한 시뮬레이션은 오히려 "가짜"로 보인다.

3. **수치 데이터 표시:** 단순 애니메이션과 공학적 시뮬레이션을 구분하는 핵심은 **정량적 데이터**다. 각 센서의 측정 거리(m), 감지 신뢰도(%), 반응 지연(ms) 등을 실시간으로 표시.

4. **센서 파라미터 패널:** LiDAR 채널 수, 회전 속도, 포인트 밀도 등 실제 사양을 UI에 표시하여 "이 시뮬레이션은 VLP-16 사양을 기반으로 한다"를 명시.

### 4.2 Canvas 2D 렌더링 성능 (중간)

LiDAR 포인트 클라우드를 Canvas 2D로 렌더링할 때:
- 360도 × 0.4도 해상도 = 900 레이/프레임
- 16채널 = 14,400 포인트/프레임 (full spec)
- Canvas 2D에서 14,400개 포인트를 매 프레임 그리면 성능 저하 가능

**완화:** PoC에서는 단일 수평면(1채널)만 시각화하고, 나머지 채널은 "데이터 수집 중"으로 표시. 900 포인트/프레임은 Canvas 2D로 충분.

### 4.3 프론트엔드 vs 백엔드 시뮬레이션 아키텍처 (Q2, 높음)

**권장: 프론트엔드(JavaScript) 단독 처리**

이유:
- 센서 시뮬레이션은 실시간 인터랙션이 핵심이므로 네트워크 지연(latency)이 없어야 한다
- 레이 캐스팅 계산은 삼각함수(sin/cos)와 교차 판정(intersection test)이 주이며, JavaScript에서 충분히 빠르다
- Python 백엔드는 이미 도메인 1~3의 AI 추론으로 CPU를 사용 중이므로, 도메인 4까지 맡기면 부하 과다
- 프론트엔드에서 시뮬레이션 → 이벤트 발생 시에만 백엔드에 POST (경고 이벤트 전송)

### 4.4 초음파 센서의 물리적 한계 미인지 (중간)

초음파 센서의 알려진 한계:
- **온도 의존성:** 음속은 온도에 따라 변함 (20°C: 343m/s, 0°C: 331m/s)
- **빔 폭:** 15도 콘이므로 각도 분해능이 매우 낮음 — "무언가 있다"는 알지만 "어디에 있는지"는 부정확
- **크로스토크:** 인접 센서끼리 간섭
- **표면 각도:** 객체 표면이 센서를 향해 기울어져 있으면 반사가 빗나감

이러한 한계를 시뮬레이션에 반영하면 오히려 **신뢰도가 올라간다** — "우리는 센서의 한계를 알고 있고, 그래서 센서 융합을 한다"는 메시지를 줄 수 있기 때문이다.

## 5. 시뮬레이션 설계 제안

### 5.1 LiDAR 시뮬레이션 모델

**기준 센서:** Velodyne VLP-16 (Puck)

| 파라미터 | 값 | 시뮬레이션 적용 |
|----------|-----|----------------|
| 채널 수 | 16 | PoC: 1채널 (수평면) 시각화, 나머지는 수치만 |
| 수평 FOV | 360도 | 전체 방사형 레이 캐스팅 |
| 수평 각도 해상도 | 0.1~0.4도 | PoC: 1.0도 (360 레이/회전) — 시각적 밀도와 성능 균형 |
| 수직 FOV | ±15도 (30도 전체) | PoC: 수평면만 (높이 0.5m에서 수평 스캔) |
| 최대 거리 | 100m | PoC: 15m (차량 주변 시각화 범위) |
| 포인트 밀도 | ~300,000 pts/sec | PoC: 360 pts/frame × 10Hz = 3,600 pts/sec |
| 측정 노이즈 | ±3cm | 가우시안 노이즈 모델: `σ = 0.03m` |
| 드롭아웃 확률 | ~2% | 랜덤하게 포인트 누락 |
| 회전 속도 | 5~20Hz | PoC: 10Hz (100ms/회전) |

**레이 캐스팅 알고리즘:**

```
for angle = 0 to 359 step 1.0:
    ray_origin = lidar_position  // 차량 지붕 중앙
    ray_direction = (cos(angle), sin(angle))

    // 모든 장애물과 교차 판정
    closest_hit = MAX_RANGE  // 15m
    for obstacle in obstacles:
        hit_distance = ray_intersect(ray_origin, ray_direction, obstacle)
        if hit_distance < closest_hit:
            closest_hit = hit_distance

    // 노이즈 추가
    measured_distance = closest_hit + gaussian_noise(0, 0.03)

    // 드롭아웃 시뮬레이션
    if random() > 0.02:
        point_cloud.add(angle, measured_distance)
```

**교차 판정:** 어린이 객체는 타원(ellipse)로 모델링. 장애물(주차 차량, 기둥)은 직사각형(AABB).

### 5.2 초음파 센서 시뮬레이션 모델

**기준 센서:** HC-SR04급 산업용 초음파 센서

| 파라미터 | 값 | 시뮬레이션 적용 |
|----------|-----|----------------|
| 주파수 | 40kHz | 표시 전용 (시뮬레이션에 직접 영향 없음) |
| 빔 폭 | 15도 (반각 7.5도) | 콘(cone) 형태 감지 영역 시각화 |
| 최소 거리 | 2cm | 0.02m 이하 무시 |
| 최대 거리 | 4m (실용 범위) | 4m 초과 시 "무응답" 처리 |
| 측정 노이즈 | ±1cm | 가우시안 노이즈: `σ = 0.01m` |
| 샘플링 주기 | 50ms (20Hz) | 초음파 왕복 시간 기반 |
| 사각지대 | 0~2cm | 최소 감지 거리 존재 |
| 센서 수 | 12개 | 전방 4, 후방 4, 측면 각 2 |

**센서 배치 (차량 중심 기준):**

```
차량 치수: 폭 2.0m (W), 길이 7.0m (L)
차량 중심 = (0, 0)

전방 센서 (y = +3.5m):
  UF1: (-0.75, +3.5) → 방향 90도, 빔 ±7.5도
  UF2: (-0.25, +3.5) → 방향 90도, 빔 ±7.5도
  UF3: (+0.25, +3.5) → 방향 90도, 빔 ±7.5도
  UF4: (+0.75, +3.5) → 방향 90도, 빔 ±7.5도

후방 센서 (y = -3.5m):
  UR1: (-0.75, -3.5) → 방향 270도, 빔 ±7.5도
  UR2: (-0.25, -3.5) → 방향 270도, 빔 ±7.5도
  UR3: (+0.25, -3.5) → 방향 270도, 빔 ±7.5도
  UR4: (+0.75, -3.5) → 방향 270도, 빔 ±7.5도

좌측 센서 (x = -1.0m):
  UL1: (-1.0, +1.5) → 방향 180도, 빔 ±7.5도
  UL2: (-1.0, -1.5) → 방향 180도, 빔 ±7.5도

우측 센서 (x = +1.0m):
  UR1: (+1.0, +1.5) → 방향 0도, 빔 ±7.5도
  UR2: (+1.0, -1.5) → 방향 0도, 빔 ±7.5도
```

**거리 측정 알고리즘:**

```
for sensor in ultrasonic_sensors:
    // 콘 내부의 가장 가까운 객체 탐색
    min_distance = MAX_RANGE  // 4m
    for obstacle in obstacles:
        if is_within_cone(sensor.position, sensor.direction, sensor.beam_width, obstacle):
            dist = distance(sensor.position, obstacle.nearest_point)
            if dist < min_distance and dist >= MIN_RANGE:
                min_distance = dist

    // 노이즈 추가
    if min_distance < MAX_RANGE:
        measured = min_distance + gaussian_noise(0, 0.01)
        sensor.reading = clamp(measured, MIN_RANGE, MAX_RANGE)
    else:
        sensor.reading = null  // 감지 없음
```

### 5.3 센서 융합 (Sensor Fusion) 설계

**권장 방식: 규칙 기반 결정 융합 (Rule-based Decision Fusion)**

칼만 필터는 단일 객체 추적에 적합하나, PoC에서는 **규칙 기반 융합**이 심사위원에게 설명하기 쉽고 구현/디버깅이 용이하다.

**융합 로직:**

```
function fuseSensors(lidar_detections, ultrasonic_readings):
    fused_objects = []

    for lidar_obj in lidar_detections:
        // 같은 방향의 초음파 센서 확인
        matching_ultrasonic = find_nearest_ultrasonic(lidar_obj.angle)

        if matching_ultrasonic and matching_ultrasonic.reading != null:
            // 두 센서 모두 감지 → 높은 신뢰도
            fused_distance = weighted_average(
                lidar_obj.distance, weight=0.7,   // LiDAR 더 정확
                matching_ultrasonic.reading, weight=0.3
            )
            confidence = 0.95
            source = "FUSION"
        else:
            // LiDAR만 감지 → 중간 신뢰도
            fused_distance = lidar_obj.distance
            confidence = 0.70
            source = "LIDAR_ONLY"

        fused_objects.append({
            distance: fused_distance,
            angle: lidar_obj.angle,
            confidence: confidence,
            source: source
        })

    // 초음파만 감지한 객체 (LiDAR 놓침 — 매우 가까운 객체)
    for sensor in ultrasonic_sensors:
        if sensor.reading != null and not already_fused(sensor):
            fused_objects.append({
                distance: sensor.reading,
                angle: sensor.direction,
                confidence: 0.60,
                source: "ULTRASONIC_ONLY"
            })

    return fused_objects
```

**경고 결정 로직:**

```
function determineAlertLevel(fused_object):
    d = fused_object.distance
    c = fused_object.confidence

    if d <= 1.5 and c >= 0.50:
        return DANGER   // 빨간색, 경보음, "차량 이동 금지"
    else if d <= 3.0 and c >= 0.60:
        return WARNING  // 주황색, 경고음
    else if d <= 5.0 and c >= 0.50:
        if fused_object.source == "LIDAR_ONLY":
            return CAUTION  // 노란색 (LiDAR만 감지 — 초음파 범위 밖)
        else:
            return CAUTION
    else:
        return CLEAR
```

**신뢰도 가중치 근거:**
- LiDAR 가중 0.7: 각도/거리 정밀도 높음, 원거리 감지 가능
- 초음파 가중 0.3: 근거리에서 보완적, 그러나 각도 해상도 낮음
- 양 센서 동시 감지 시 신뢰도 0.95: 독립적 두 센서가 동일 객체 확인

### 6.1 렌더링 기술 선택

**권장: HTML5 Canvas 2D**

| 기술 | 장점 | 단점 | 판정 |
|------|------|------|------|
| Canvas 2D | 의존성 없음, 빠른 구현, 충분한 성능 | 3D 표현 불가 | **권장** |
| Three.js 3D | 인상적, 포인트 클라우드 자연스러움 | 의존성 추가, 복잡도 3배, 비기술자에게 오히려 혼란 | 비권장 |
| SVG | 선명한 벡터 | 수백 개 포인트에서 성능 저하 | 비권장 |
| WebGL 직접 | 최고 성능 | 구현 복잡도 극히 높음 | 비권장 |

Canvas 2D가 적절한 이유:
1. Bird's Eye View는 본질적으로 **2D 탑다운 뷰**이므로 3D가 오히려 불필요한 복잡도를 추가
2. 심사위원은 3D 회전/줌보다 정적인 2D 조감도를 더 직관적으로 이해
3. 360개 레이 포인트 + 12개 초음파 콘 = Canvas 2D로 60fps 충분
4. 별도 라이브러리 없이 순수 JavaScript로 구현 가능

### 6.3 시각 요소 상세

**LiDAR 포인트 클라우드:**
- 360개 레이를 차량 중심에서 방사형으로 표시
- 레이 자체: 반투명 밝은 초록색 선 (`rgba(0, 255, 100, 0.15)`)
- 반사 포인트: 밝은 초록색 원 (반경 2px), 객체에 부딪힌 위치
- 최대 거리 도달 레이: 더 투명하게 (미감지)
- 노이즈: 포인트 위치가 프레임마다 약간 흔들림 (±2px)

**초음파 콘:**
- 각 센서에서 15도 부채꼴 영역
- 감지 없음: 반투명 파란색 (`rgba(0, 150, 255, 0.08)`)
- 감지 중: 거리에 따라 색상 변화:
  - 4~3m: 노란색 (`rgba(255, 220, 0, 0.15)`)
  - 3~1.5m: 주황색 (`rgba(255, 140, 0, 0.20)`)
  - 1.5m 이하: 빨간색 (`rgba(255, 0, 0, 0.25)`)

**어린이 객체:**
- 타원형 (폭 35px ≈ 35cm, 높이 20px ≈ 20cm — 탑다운)
- 파란색 윤곽, "어린이" 라벨
- 이동 경로: 점선으로 궤적 표시
- 접근 시: 빨간색 하이라이트 + 물결 효과(pulse)

**경고 단계별 시각 효과:**

| 단계 | 거리 | 시각 효과 | 음향 |
|------|------|----------|------|
| CLEAR | >5m | 초록색 테두리, 정상 상태 | 없음 |
| CAUTION (주의) | ≤5m | 노란색 동심원 펄스, 해당 방향 하이라이트 | 낮은 "띵" 1회 |
| WARNING (경고) | ≤3m | 주황색 깜빡임(1Hz), 센서 방향 화살표 | 중간 경고음 반복 |
| DANGER (위험) | ≤1.5m | 빨간색 깜빡임(2Hz), 화면 테두리 빨간색, "정지" 오버레이 | 높은 경보음 연속 + 음성 "차량 이동을 금지합니다" |

**차량 표현:**
- 탑다운 뷰 직사각형 (스케일: 1m = ~40px → 차량 80×280px)
- 노란색 배경 + 검은 테두리
- "어린이 보호" 텍스트 중앙
- 전방에 방향 삼각형
- 각 센서 위치에 작은 원형/부채꼴 아이콘

**입증 포인트:**
- 센서 융합의 핵심 가치: 단일 센서 성능 저하 시 다른 센서가 보완
- 악천후에서도 안전 시스템 동작 가능
- 노이즈/드롭아웃의 시각적 표현 → 공학적 시뮬레이션 입증

## 8. 실제 센서 사양과의 Gap 분석

### 8.1 규제 심사위원 예상 질문 및 답변

**Q1: "이 시뮬레이션이 실제 LiDAR 센서와 얼마나 유사합니까?"**

> **A:** 본 시뮬레이션은 Velodyne VLP-16 LiDAR의 실제 사양(16채널, 360도 FOV, 100m 범위, 0.4도 각도 해상도)을 기반으로 합니다. PoC에서는 2D 수평면 1채널을 시각화하되, 레이 캐스팅 알고리즘, 측정 노이즈(±3cm 가우시안), 포인트 드롭아웃(2%) 등 실제 센서 특성을 반영했습니다. 실 차량 탑재 시에는 16채널 전체를 활용하여 높이 정보까지 포함한 3D 감지가 가능합니다.

**Q2: "초음파 센서의 감지 범위가 4m인데, 5m에서 경고를 어떻게 하나요?"**

> **A:** 5m 주의 경고는 LiDAR(100m 범위)가 담당하고, 초음파는 4m 이하에서 보조 확인 역할을 합니다. 이것이 바로 **센서 융합**의 핵심입니다. LiDAR는 원거리에서 먼저 감지하고, 초음파는 근거리에서 추가 확인하여 신뢰도를 높입니다. 두 센서의 감지 범위가 다르기 때문에 상호 보완적입니다.

**Q3: "비가 오거나 안개가 끼면 LiDAR가 동작합니까?"**

> **A:** 악천후 시 LiDAR 성능은 저하됩니다(범위 최대 30% 감소, 노이즈 증가). 시나리오 S5에서 이를 시뮬레이션했으며, 바로 이런 상황에서 초음파 센서가 중요합니다. 초음파는 날씨 영향을 거의 받지 않으므로, LiDAR 성능이 떨어질 때 초음파가 근거리 감지를 보완합니다. 이것이 **단일 센서가 아닌 융합 시스템**을 설계한 이유입니다.

**Q4: "어린이가 차량 바로 옆에 숨어 있으면 감지할 수 있나요?"**

> **A:** LiDAR는 차량 지붕에 설치되므로 차량 바로 옆 근접 영역(~0.5m 이내)은 수직 각도 한계로 사각지대가 될 수 있습니다. 이를 보완하기 위해 초음파 센서를 범퍼 높이(0.5m)에 12개 배치하여 근접 사각지대를 커버합니다. 시나리오 S4에서 장애물 뒤 은폐 상황도 시뮬레이션했으며, 이 경우 물리적으로 센서 파가 도달할 수 없으므로 솔직하게 "미감지"로 표시한 뒤, 객체가 출현하는 즉시 감지하는 것을 보여줍니다.

**Q5: "시뮬레이션의 정확도를 어떻게 검증했습니까?"**

> **A:** 본 PoC는 기술 원리의 타당성 입증이 목적이므로, 실제 센서 데이터와의 정량적 비교 검증은 다음 단계(실증 사업)에서 수행합니다. 다만 시뮬레이션에 사용된 모든 파라미터(감지 범위, 노이즈 모델, 각도 해상도)는 상용 센서 데이터시트(Velodyne VLP-16, 산업용 초음파 센서)를 기반으로 설정했으며, 이는 센서 파라미터 패널에서 실시간으로 확인할 수 있습니다.

**Q6: "실제 차량에 몇 개의 센서를 설치할 예정입니까? 비용은?"**

> **A:** LiDAR 1대(차량 지붕 중앙) + 초음파 12개(전후좌우 범퍼)를 기본 구성으로 합니다. Velodyne VLP-16 기준 약 400~500만원, 초음파 센서 12개 약 30만원, 총 하드웨어 비용 약 500만원 수준입니다. 양산 시에는 Solid-state LiDAR(예: Livox Mid-70)로 대체하면 100~200만원대로 절감 가능합니다.
