### ITEM-A01: SOS/긴급 버튼 — PASS

112/119 긴급전화 연결은 법적 제한 없음. 오히려 아동복지법상 아동 안전 의무와 부합.

**추천 사항 (법적 의무는 아님):**
- SOS 이벤트 감사 로그를 최소 3년간 보관 (민법 제766조 불법행위 손해배상 소멸시효)
- SOS 이벤트 발생 시 관할 경찰서 자동 통보 기능은 경찰청 협의 후 추가 검토

### ITEM-A02: 기사/차량 정보 학부모 앱 노출 — PASS

기사 전화번호 마스킹 처리(`010-****-5678`)가 기획에 포함되어 있어 개인정보보호법 제17조(목적 외 이용 금지) 및 제18조(제3자 제공 제한)에 부합.

**확인 사항:**
- 기사 전화번호 마스킹은 서버 측에서 수행해야 함 (클라이언트 마스킹만으로는 불충분)
- 기획서에 서버 측 마스킹 명시되어 있어 적합

### ITEM-A03: 잔류 학생 확인 — NEEDS_AMENDMENT

**관련 법률:** 도로교통법 제53조의3 (세림이법)

**적합 부분:**
- 3단계 체크리스트(좌석/트렁크/잠금)는 세림이법 취지에 부합
- 서버 기록 저장으로 증빙 가능

**수정 필요 사항:**

1. **세림이법 증빙 보관 기간 명시 필요**
   - 현재 기획서에 체크리스트 기록 보관 기간 정의 없음
   - 도로교통법 시행규칙상 어린이 통학버스 운행 기록은 **최소 1년간** 보관 의무
   - 기획서에 `VehicleClearance` 레코드 보관 기간 1년 이상 명시 필요

2. **미완료 시 제재 조치 구체화 필요**
   - 현재: "미완료 시 해당 차량의 운행 종료 불가"
   - 추가 필요: 미완료 상태 30분 경과 시 관제센터 자동 에스컬레이션 + 112 자동 연락 검토
   - 세림이법 제정 배경(잠든 아이 방치 사고)을 고려하면, 단순 차단보다 적극적 대응 프로세스 필요

3. **안전도우미 확인 절차 명시 필요**
   - 세림이법상 동승보호자 의무가 있으므로, 차량 점검은 기사 + 안전도우미 **이중 확인** 필요
   - 기획서에 안전도우미 확인 서명/체크도 추가 권고

### ITEM-A07: OTP 보안 강화 — NEEDS_AMENDMENT

**관련 법률:** 개인정보보호법 제29조 (안전성 확보 조치), 개인정보의 안전성 확보조치 기준(고시) 제5조

**적합 부분:**
- `secrets` 모듈 전환: 안전성 확보조치 기준 제7조(암호화) 요건 충족
- 5회 실패 잠금: 접근통제 요건 충족

**수정 필요 사항:**

1. **OTP 유효시간 명시 필요**
   - 현재 기획서에 OTP TTL 명시 없음
   - 안전성 확보조치 기준상 인증 코드는 **3분 이내** 만료 권고
   - 기획서에 OTP TTL = 180초 (3분) 명시 필요

2. **OTP 발송 기록 보관**
   - OTP 발송/검증 이벤트를 감사 로그에 기록해야 함 (안전성 확보조치 기준 제8조 - 접속기록 보관)
   - 접속기록(인증 시도) **최소 1년** 보관 의무 (5만 명 이상 시 2년)

3. **잠금 해제 방법 명시**
   - 15분 자동 해제 외에 본인 확인 후 수동 해제 절차 필요 여부 검토
   - 계정 탈취 시나리오 대비

### ITEM-A08: IDOR 수정 — NEEDS_AMENDMENT

**관련 법률:** 개인정보보호법 제29조, 안전성 확보조치 기준 제5조 (접근통제)

**적합 부분:**
- 학부모 소유권 확인 (guardian_id 대조): 접근통제 요건 충족
- 역할별 접근 범위 차등 설계: 적합

**수정 필요 사항:**

1. **접근통제 대상 API 확대 필요**
   - 기획서에 `get_student`, `get_consent`, `list_enrollments` 3개 API만 명시
   - 추가로 IDOR 취약점 확인이 필요한 API:
     - `GET /schedules/daily/{instance_id}` — 스케줄 인스턴스 개별 조회
     - `PATCH /students/{student_id}` — 학생 정보 수정
     - `GET /compliance/documents/{doc_id}` — 컴플라이언스 문서 조회
     - `GET /notifications` — 알림 조회 (타인 알림 접근 가능성)
   - **전체 API 대상 접근통제 감사(audit)** 수행 후 누락 API 보완 필요

2. **접근 실패 로깅**
   - 403 Forbidden 응답 시 해당 시도를 감사 로그에 기록해야 함
   - 반복적 IDOR 시도는 보안 이벤트로 분류하여 관제센터에 알림

3. **학원 관리자 소속 확인 로직 구체화**
   - 기획서 코드 주석: `# ACADEMY_ADMIN: 학원 소속 확인 (enrollment 테이블 조회)` — 구체적 구현 미명시
   - enrollment 테이블의 academy_id와 현재 사용자의 academy_id 교차 검증 로직 명확히 기술 필요

### ITEM-A09: WebSocket 차량 위치 스트림 인가 — NEEDS_AMENDMENT

**관련 법률:** 위치정보법 제15조 (개인위치정보 수집 동의), 제16조 (수집 기록), 제18조 (제3자 제공)

**적합 부분:**
- 역할별 차량 접근 권한 설계: 위치정보 접근통제 원칙에 부합
- 학부모는 자녀 탑승 차량만 추적: 최소 수집/이용 원칙 충족

**수정 필요 사항:**

1. **위치정보 수집/이용/제공 기록 연동 필요**
   - 위치정보법 제16조: "개인위치정보의 수집·이용·제공 사실 확인자료를 자동으로 기록·보존"
   - WebSocket 연결 성공 시 (=위치정보 제공 시점), 다음 기록을 별도 테이블에 저장:
     - 위치정보 주체 (아동/보호자)
     - 이용 목적 (안전 모니터링)
     - 제공받는 자 (접속한 사용자 정보)
     - 접속 일시
   - 이 기록은 **6개월간** 보관 의무 (위치정보법 제24조)

2. **위치정보 동의 미부여자 차단 연동**
   - `check_vehicle_access` 함수에서 학부모의 위치정보 동의 여부도 확인해야 함
   - 동의(`consent_scope.location_tracking = true`) 미부여 시 WebSocket 연결 거부
   - ITEM-A14의 동의 프로세스와 연동 필수

3. **기사/안전도우미의 위치정보 제공 동의**
   - 기사/안전도우미는 GPS 데이터를 생성(제공)하는 주체
   - 이들의 개인위치정보 수집에 대한 동의도 필요 (고용 계약 또는 이용약관에 포함 가능)
   - 기획서에 기사/안전도우미 위치정보 동의 절차 추가 필요

### ITEM-A11: 운전자 자격 검증 모델 — NEEDS_AMENDMENT

**관련 법률:** 도로교통법 제52조 제2항, 아동복지법 제29조의3, 도로교통법 시행규칙 제35조

**적합 부분:**
- DriverQualification 모델 설계: 핵심 필드(면허번호, 면허유형, 범죄경력, 안전교육) 포함
- 자격 미충족 배정 차단 로직: 법적 요건에 부합

**수정 필요 사항:**

1. **면허번호 암호화 저장 필수**
   - 운전면허번호는 개인정보보호법 시행령 제19조의 **고유식별정보**
   - 고유식별정보는 반드시 **암호화 저장** 의무 (안전성 확보조치 기준 제7조 제1항)
   - 기획서에 `license_number` 필드의 AES-256 암호화 저장 명시 필요
   - 이전 법규 리뷰(P17 P-03)에서도 지적: 현재 `encrypt_value` 유틸은 존재하나 미사용

2. **범죄경력 조회 절차 구체화 필요**
   - 아동복지법 제29조의3: 아동관련기관 종사자는 **성범죄/아동학대 범죄경력** 조회 의무
   - 범죄경력 조회는 관할 경찰서를 통해 진행 (플랫폼이 직접 조회 불가)
   - 기획서에 범죄경력 조회 프로세스 명시 필요:
     - 학원장이 관할 경찰서에 조회 요청 → 결과 서류를 플랫폼에 업로드
     - `ComplianceDocument` 테이블의 `POLICE_REPORT` 타입과 `DriverQualification.criminal_check_clear` 연동
     - 조회 결과 유효기간: **1년** (매년 갱신)

3. **안전교육 이수 요건 구체화**
   - 도로교통법 시행규칙 제35조: 어린이통학버스 운전자 안전교육 **매년 3시간** 이수 의무
   - `safety_training_expiry` 필드만으로는 부족 — 교육 이수 증빙 서류 업로드 연동 필요
   - `ComplianceDocument` 테이블의 `SAFETY_TRAINING` 타입과 연동

4. **자격 만료 처리 강제성 확보**
   - 면허 만료 또는 안전교육 미갱신 시 **즉시 배차 차단** (30일 전 알림은 경고, 만료일 이후는 강제 차단)
   - 범죄경력 조회 갱신 미수행 시(1년 경과) 역시 배차 차단

### ITEM-A12: 차량 법정 필수 필드 — NEEDS_AMENDMENT

**관련 법률:** 도로교통법 제52조, 자동차관리법 시행규칙 제3조, 어린이통학버스 안전관리 강화 방안

**적합 부분:**
- 제조연도, 통학버스 신고번호, 황색 도색 확인 등 법정 필수 필드 추가: 적합
- 배차 검증 로직 포함: 적합

**수정 필요 사항:**

1. **차량 연식 기준 정정**
   - 기획서: "manufacture_year가 현재연도 - 9 이상 (9년 초과 차량 차단)"
   - 도로교통법 제52조 및 시행규칙: 어린이 통학버스의 연식 제한은 **운행 시작 후 11년** (2024년 개정)
   - SRS 자체 기준은 "3년 이내"이나, 이는 SRS 자체 목표치이므로 법적 의무와 구분 필요
   - 법적 의무: 11년 초과 시 강제 차단
   - 플랫폼 기준: 3년 이내 권고 (경고 표시), 9년 초과 시 배차 제한
   - 두 기준을 구분하여 기획서에 반영 필요

2. **보험 가입 요건 구체화**
   - 어린이 통학버스는 **대인배상 무한** + **대물배상 2천만 원 이상** 의무가입
   - 기존 `insurance_expiry` 필드에 추가로 보험 종류/보장 범위 필드 필요:
     - `insurance_type` (대인, 대물, 종합)
     - `insurance_coverage_amount` (보장 금액)
   - 보험 미가입 또는 미달 차량 배차 차단

3. **차량 정기 검사 연동**

### ITEM-A14: 14세 미만 아동 법정대리인 동의 프로세스 강화 — NEEDS_AMENDMENT

**관련 법률:** 개인정보보호법 제22조의2, 동법 시행령 제17조, 위치정보법 제18조

**적합 부분:**
- 동의 항목 필수/선택 구분 구조화: 적합
- 선택 동의 기본값 False: 적합
- 동의 철회 시 관련 기능 비활성화: 적합

**수정 필요 사항:**

1. **법정대리인 본인확인 수단 강화 필수**
   - 개인정보보호법 시행령 제17조 제1항: 법정대리인의 동의를 확인하는 방법
     - 1호: 동의 내용을 게재한 인터넷 사이트에 법정대리인이 동의 여부를 표시
     - 2호: 동의 내용이 적힌 서면을 법정대리인에게 직접 발급하거나 우편 등으로 송부하여 동의의 의사표시를 수령
     - 3호: 전화를 통하여 동의 내용을 법정대리인에게 알리고 동의를 얻거나, 전화를 통한 동의 내용 안내 후 동의 내용이 적힌 문서를 전자우편, 팩스 등으로 발송하여 동의의 의사표시를 수령
     - 4호: 그 밖에 위와 유사한 방법 (CI 연동 등)
   - 현재 기획서: 구체적 본인확인 방법 미명시
   - **단계별 접근 권고:**
     - Phase 1 (MVP): 3호 방식 — OTP 인증 + 동의서 전문 안내 + 전자서명(체크박스) → 법적 최소 요건 충족 가능
     - Phase 2 (규제 샌드박스 심사 전): 본인인증기관(CI) 연동 또는 공동인증서 → 강화된 본인확인
   - 기획서에 Phase 1 최소 요건이라도 구체적 구현 방안 명시 필요

2. **법정대리인-아동 관계 검증**
   - `guardian_id`가 실제 법정대리인인지 확인하는 절차 필요
   - 최소 요건: 가족관계증명서 또는 주민등록등본 업로드 → 관리자 확인
   - `ComplianceDocument` 테이블에 `GUARDIAN_RELATION_PROOF` 타입 추가 권고

3. **동의 철회 시 기수집 데이터 처리 명시 필요**
   - 기획서: location_tracking 철회 시 GPS 추적 비활성화 + child_info_collection 철회 시 스케줄 비활성화
   - 추가 필요: 기수집 위치정보의 처리 방법 (즉시 삭제 vs 보관기간 만료 후 삭제)
   - 개인정보보호법 제36조: 삭제 요구 시 지체 없이 파기 → **즉시 삭제** 원칙
   - 단, 위치정보법 제24조의 6개월 보관 의무와 충돌 → 수집 기록은 보관하되 원본 GPS 데이터는 삭제

4. **동의서 전문 열람 기록 저장**
   - 이전 법규 리뷰(P17 P-01)에서 지적: 동의서 전문(全文) 열람 기록 저장 필요
   - 동의 화면 진입 시각, 스크롤 완료 시각, 동의 체크 시각을 기록하여 "읽고 동의했음"의 증빙으로 활용

| 7 | 개인정보(전화번호, GPS 등) 암호화 방식은? | **부분** — P0 기획서에 암호화 적용 항목 없음 (code-hardening 별도) |
| 8 | 데이터 보존기간 경과 시 자동 파기 절차는? | **가능** (ITEM-A13 구현 후) |
| 9 | 불법 유상운송이 아닌 법적 근거는? | **불가** — 기획서 미포함 |
| 10 | 실증특례 이행 현황 모니터링 체계는? | **불가** — 기획서 미포함 |

## 종합 소견

### 법률 적합성 종합 평가

P0 기획서 27개 항목 중 **16개는 법률적으로 적합(PASS)**, **11개는 수정 필요(NEEDS_AMENDMENT)**. NEEDS_AMENDMENT 항목 중 기획서 자체의 방향은 올바르나, 법적 요건의 세부 충족을 위해 보완이 필요한 경우가 대부분.

### 핵심 법적 리스크 3가지

1. **위치기반서비스사업자 미신고 상태에서의 서비스 운영** (위치정보법 제5조의2 위반 — 벌칙: 3년 이하 징역 또는 3천만 원 이하 벌금)
   - 코드 구현과 별개로 **행정 신고 절차가 선행 또는 병행**되어야 함

2. **고유식별정보(운전면허번호) 평문 저장** (개인정보보호법 제24조 위반 — 과태료 최대 5천만 원)
   - ITEM-A11에서 DriverQualification 모델 추가 시 반드시 암호화 저장 적용

3. **동승보호자 미배치 운행 차단 미구현** (도로교통법 제53조의3 위반)
   - 규제 샌드박스 심사 핵심 항목이므로 기획서에 추가 필요

### 권고 우선순위

1. **즉시**: ITEM-A11(면허번호 암호화), ITEM-A13(위치정보사업자 신고 착수), ITEM-A14(동의 프로세스 최소요건)
2. **1주 이내**: 동승보호자 강제 배치 항목 기획서 추가, 전세버스 계약 구조 보강 항목 기획서 추가
3. **서비스 오픈 전**: 위치기반서비스사업자 신고 완료, 위치정보 이용약관 작성, 개인정보 영향평가 수행
