## 1. 요구사항 재진술

SafeWay Kids Edge AI PoC는 **규제 샌드박스 심사위원**(비기술 전문가)에게 어린이 통학차량 AI 안전 기술의 실현 가능성을 입증하기 위한 **소프트웨어 시뮬레이션 데모**이다.

핵심 요구사항:
- **4개 도메인을 4개 시나리오로 매핑한 통합 데모:** (1) 안면인식 승하차, (2) 운행 중 이상행동 감지, (3) 시동 OFF 잔류인원 감지, (4) 사각지대 어린이 감지
- **실행 환경:** Windows i5 PC + USB 웹캠 1대, GPU 없음 (CPU only)
- **출력:** 웹 브라우저 기반 실시간 데모 UI
- **시각적 완성도:** 비기술 심사위원이 직관적으로 이해하고 "이 기술이 실제로 동작한다"고 납득할 수 있는 수준
- **원클릭 실행:** batch 파일 또는 단일 명령으로 즉시 실행
- **안정성:** 최소 10분 연속 가동, 데모 중 크래시 절대 불가
- **성능 지표 실시간 시각화:** 추론 지연시간, FPS, CPU 사용률 등 신청서 목표치를 화면에 표시
- **백엔드 연동:** SafeWay Kids 백엔드로 이벤트 전송 + standalone 모드 병행

데모의 본질은 **"기술 시연"이 아니라 "규제 설득"**이다. 심사위원은 코드가 아닌 **시각적 결과물**로 판단한다.

## 3. 충돌/모순

| # | 충돌 항목 | 설명 | 권장 해결 |
|---|-----------|------|----------|
| C1 | **시나리오 ①~③은 실시간 웹캠 / ④는 소프트웨어 시뮬레이션** | ①~③은 MJPEG 비디오 스트림 기반이고 ④는 Canvas 2D 기반 → 렌더링 방식이 근본적으로 다름. 하나의 "video feed" 영역에 넣을 수 없음 | 시나리오 ④는 별도 Canvas 영역으로 분리. 메인 레이아웃을 "비디오 영역 + Canvas 영역" 또는 "탭별 전체 교체"로 설계 |
| C2 | **G3 "원클릭 실행" vs G5 "백엔드 연동"** | 원클릭 실행은 독립 실행을 암시하지만 백엔드 연동은 별도 서버 필요 → 모순 | standalone 모드를 기본으로 하되, 백엔드 감지 시 자동 연동. `--with-backend` 플래그로 명시적 선택 |
| C3 | **AC1 "95% 인식률" vs A10 "dlib 빌드 실패 가능"** | 95% 달성에는 deep embedding 필수인데, 가장 접근성 높은 face_recognition(dlib)이 Windows에서 빌드 실패 위험 높음 | InsightFace (ONNX Runtime) 을 primary로, MediaPipe FaceMesh를 fallback으로. dlib은 시도하지 않기를 권장 |
| C4 | **NG7 "단일 유저" vs 프로젝터 발표 환경** | 단일 브라우저라고 했지만, 실제로는 발표자 PC + 프로젝터 연결. 프로젝터 해상도가 다를 수 있음 | 반응형 레이아웃 필수. 1280x720, 1920x1080 모두 대응 |
| C5 | **"10분 연속 가동" vs MJPEG 스트리밍** | MJPEG는 브라우저에서 장시간 스트리밍 시 메모리 누수 발생 가능 (img 태그의 multipart 스트림) | WebSocket 기반 프레임 전송으로 전환하거나, 주기적 img.src 리로드 방어 코드 추가 |

## 4. 기술 리스크

### 4.1 높은 리스크

| # | 리스크 | 상세 | 대응 |
|---|--------|------|------|
| TR1 | **Flask MJPEG의 UX 한계** | 현재 구현은 `<img src="/video_feed">`로 MJPEG 스트림을 표시. 이 방식은: (a) 프레임 위에 HTML 오버레이를 올릴 수 없음 (b) Canvas와 혼합 렌더링 불가 (c) 지연(latency) 누적으로 시간이 지날수록 느려짐 (d) 브라우저마다 동작이 다름 (Safari 이슈) | WebSocket으로 JPEG 프레임을 전송하고 Canvas에 그리는 방식으로 전환 |
| TR2 | **4개 시나리오 통합의 복잡도** | 시나리오 ①~③은 동일 카메라 파이프라인이지만 ④는 완전히 다른 렌더링. 단일 페이지에 4개를 넣으면 상태 관리가 복잡해짐 | 스텝 위저드 패턴: 한 번에 하나의 시나리오만 활성화. 각 시나리오는 독립 모듈로 격리 |
| TR3 | **CPU 멀티 모델 동시 추론 부하** | YOLOv8n + MediaPipe Pose + MediaPipe Face를 i5에서 동시 실행하면 프레임 드롭 + UI 응답 지연 가능 | 시나리오 전환 시 이전 모델을 언로드. 한 시나리오당 필요한 모델만 로드 |
| TR4 | **안면인식 정확도 95% 달성** | 히스토그램 매칭으로는 불가능. InsightFace ONNX는 CPU에서 ~50ms/face로 가능하나 설치 복잡도 있음 | InsightFace (onnxruntime-cpu) 기반 구현. pip install로 해결되며 dlib 빌드 불필요 |

### 4.2 중간 리스크

| # | 리스크 | 상세 | 대응 |
|---|--------|------|------|
| TR5 | **사각지대 시뮬레이션의 신뢰성** | Canvas 2D 애니메이션이 "장난감"처럼 보일 수 있음 | 물리 기반 시뮬레이션 + 실제 LiDAR 데이터 형식의 포인트 클라우드 렌더링. 전문적인 색상 스킴(열화상 팔레트) 사용 |
| TR6 | **Windows Python 환경 설정** | PyTorch CPU, ultralytics, mediapipe, onnxruntime 등 패키지 충돌 가능 | requirements.txt를 정확히 버전 고정. `start.bat`에서 venv 생성 + 자동 설치 |
| TR7 | **음성 경고 브라우저 정책** | Chrome은 사용자 인터랙션 없이 오디오 자동재생 차단 | 데모 시작 시 "시작" 버튼 클릭 필수 (이 인터랙션이 오디오 컨텍스트를 활성화) |

## 5. UI 프레임워크 결정 (Q4 응답)

### 5.1 옵션별 분석

#### 옵션 A: Flask + MJPEG (현재 구현)

| 장점 | 단점 |
|------|------|
| 이미 동작하는 코드 존재 | MJPEG 위에 HTML 오버레이 불가 |
| 의존성 최소 | 사각지대 Canvas와 혼합 아키텍처 필요 |
| 서버 사이드 렌더링으로 단순 | 장시간 스트리밍 시 메모리 누수 위험 |
| | 프레임 위에 성능 지표 오버레이가 CV2 텍스트로만 가능 → 시각적 품질 낮음 |
| | 실시간 양방향 통신 없음 (polling만 가능) |

**결론: 규제 심사 수준의 시각적 완성도 달성 불가. 탈락.**

#### 옵션 B: Flask + WebSocket (Socket.IO)

| 장점 | 단점 |
|------|------|
| Flask 기반 유지 → 기존 코드 활용 가능 | Socket.IO 추가 의존성 |
| WebSocket으로 실시간 양방향 통신 | 프론트엔드 UI를 여전히 직접 작성해야 함 |
| JPEG 프레임을 base64로 전송 → Canvas에 렌더링 가능 | Jinja2 템플릿 내 JS가 복잡해짐 |
| Canvas 오버레이 가능 (성능 지표, 바운딩 박스 등) | 대규모 UI 구축 시 관리 어려움 |
| 프레임 드롭 제어 가능 (클라이언트 측) | |

**결론: 실용적 선택지. 기존 코드를 최대한 활용하면서 UX를 크게 개선 가능.**

#### 옵션 C: Gradio

| 장점 | 단점 |
|------|------|
| AI 데모 특화, 빠른 프로토타이핑 | 커스텀 UI 자유도 극히 낮음 |
| 이미지/비디오 컴포넌트 내장 | 4개 시나리오 통합 탭 UI 커스터마이징 제한 |
| 공유 URL 자동 생성 | 사각지대 Canvas 시뮬레이션 통합 어려움 |
| | "Gradio 느낌"이 나서 전문성이 떨어져 보일 수 있음 |
| | 성능 대시보드 커스터마이징 한계 |

**결론: 빠르게 만들 수 있지만 규제 심사용 "전문적인 제품" 느낌 달성 어려움. 탈락.**

#### 옵션 D: Streamlit

| 장점 | 단점 |
|------|------|
| 대시보드 특화, 차트/지표 내장 | 비디오 스트리밍 성능 매우 낮음 |
| Python으로만 UI 작성 | 실시간 갱신에 전체 리렌더링 발생 |
| | 사각지대 Canvas 통합 극히 어려움 |
| | "데이터 분석 도구" 느낌 → 제품 느낌 없음 |

**결론: 비디오 스트리밍 + Canvas 시뮬레이션이라는 요구사항에 부적합. 탈락.**

#### 옵션 E: React SPA + FastAPI

| 장점 | 단점 |
|------|------|
| 최고 수준의 UI 자유도 | 구현 복잡도 매우 높음 |
| 컴포넌트 기반 아키텍처로 4개 시나리오 관리 용이 | 빌드 도구 체인 추가 (Node.js, npm, Vite) |
| 풍부한 UI 라이브러리 생태계 | 원클릭 실행이 복잡해짐 (Python + Node.js) |
| 성능 대시보드, 차트, 애니메이션 자유자재 | "일반 사무용 PC"에서 Node.js까지 설치 부담 |
| 기존 프로젝트(web/)에서 React 경험 있음 | 개발 기간 가장 길음 |

**결론: 최고 품질 가능하나 복잡도와 배포 부담이 너무 큼. 조건부 고려.**

### 5.2 최종 추천: Flask + Socket.IO + 정적 프론트엔드 (옵션 B 강화)

**추천 아키텍처:**

```
Python (Flask + Flask-SocketIO)
├── 백엔드: AI 추론 + 이벤트 처리 + 시스템 모니터링
├── WebSocket: JPEG 프레임 + 성능 데이터 + 이벤트 실시간 푸시
└── 정적 프론트엔드: HTML + CSS + Vanilla JS (빌드 도구 없음)
    ├── Canvas: 비디오 프레임 렌더링 + 오버레이
    ├── Canvas: 사각지대 시뮬레이션
    ├── Dashboard: 성능 지표 (CSS Grid + SVG 차트)
    └── Audio: Web Audio API 경고음
```

**선택 이유:**
1. **Python만으로 완결** — Node.js 설치 불필요, `pip install` + `python main.py`로 완료
2. **기존 Flask 코드 활용** — AI 추론 파이프라인, API 라우트, 이벤트 전송 코드 재사용
3. **Canvas 기반 렌더링** — 비디오 프레임 위에 바운딩 박스, 성능 지표, 경고 오버레이를 HTML/CSS로 자유롭게 배치 가능
4. **사각지대 시뮬레이션과 자연스러운 통합** — 동일한 Canvas API로 비디오 + 시뮬레이션 모두 처리
5. **빌드 도구 없음** — 정적 HTML/CSS/JS 파일을 Flask가 서빙. Vite, Webpack 불필요
6. **WebSocket으로 실시간성 확보** — polling 대신 push. 프레임 드롭 제어, 이벤트 즉시 전달

**추가 의존성:** `flask-socketio`, `python-socketio`, `eventlet` (or `gevent`) — 모두 pip install 가능

## 8. 백엔드 연동 vs 독립 실행 (Q7 응답)

### 결정: 독립 실행(Standalone)을 기본 모드로, 백엔드 연동을 옵션으로

**이유:**
1. 데모 현장에서 백엔드 서버가 실행 중이라는 보장이 없음
2. 네트워크 이슈로 백엔드 연결이 끊기면 데모 전체가 실패할 수 있음
3. 규제 심사의 핵심은 "AI 추론 능력"이지 "백엔드 통합"이 아님
4. 백엔드 연동은 "보너스 시연" 항목으로 위치

### 구현 설계

```python
# config.py
STANDALONE_MODE = True  # 기본값: 독립 실행
BACKEND_URL = "http://localhost:8000"  # 백엔드 주소 (연동 시)
BACKEND_AUTO_DETECT = True  # 시작 시 백엔드 자동 감지

# 시작 시 자동 감지 로직:
# 1. BACKEND_URL로 GET /api/health 호출 (timeout 2초)
# 2. 응답 성공 → STANDALONE_MODE = False, UI에 "백엔드 연동" 표시
# 3. 응답 실패 → STANDALONE_MODE = True, UI에 "독립 실행" 표시
# 4. 실행 중 연결 끊김 → 자동으로 standalone fallback (UI에 알림)
```

### UI 표시

```
백엔드 연동 시:  [●] Backend Connected — 이벤트 실시간 전송 중
독립 실행 시:    [○] Standalone Mode — 이벤트는 로컬 로그에 기록
```

### 이벤트 처리 분기

| 동작 | 백엔드 연동 모드 | Standalone 모드 |
|------|---------------|----------------|
| 안면인식 이벤트 | POST /api/v1/edge → 학부모 푸시 | 로컬 이벤트 로그에 기록 + UI 토스트 "학부모 알림 전송 (시뮬레이션)" |
| 이상행동 경고 | POST → 관리자 알림 | 로컬 기록 + UI 경고 표시 |
| 잔류인원 감지 | POST → 긴급 알림 | 로컬 기록 + UI 긴급 경고 |
| 사각지대 감지 | POST → 차량 정지 신호 | 로컬 기록 + UI 경고 |

| 안정성 | 75/100 | MJPEG 메모리 누수 + 멀티 모델 동시 추론 부하 → WebSocket 전환 필요 |
| 원클릭 실행 | 70/100 | start.bat 설계는 있으나 PyTorch CPU 설치 시간(5~10분), 디스크 사용량(~2GB) 미리 안내 필요 |

### 리스크 요약

1. **가장 큰 리스크:** UI 시각적 완성도. 현재 코드의 UI는 규제 심사에 적합하지 않음. 리디자인에 상당한 공수가 필요
2. **두 번째 리스크:** 안면인식 라이브러리 전환. 히스토그램 → InsightFace 전환은 파이프라인 재설계 수준
3. **세 번째 리스크:** i5 CPU에서의 추론 성능. 멀티 모델 동시 로드 시 메모리 + CPU 부하

### 승인 조건

본 Requirement Brief는 프로젝트 범위, 도메인 분류, 시나리오 정의, 기술 리스크 식별이 충실하게 이루어져 있다. 다만, 규제 심사 수준의 데모로 발전시키기 위해 다음 조건이 충족되어야 한다:

#### 필수 조건 (Must)

1. **UI 프레임워크 결정:** Flask + Socket.IO + Canvas 기반 아키텍처로 확정 (MJPEG 단독 사용 탈피)
2. **안면인식 라이브러리 확정:** InsightFace (ONNX Runtime)를 primary로 채택. dlib/face_recognition은 시도하지 않을 것
3. **Standalone 모드 설계 반영:** 백엔드 없이 독립 동작하는 모드를 기본으로, 백엔드 자동 감지를 옵션으로
4. **데모 실패 방지 설계 추가:** 사전 녹화 fallback, 에러 복구 UI, 로딩 화면 등 "데모는 절대 실패하면 안 된다"는 원칙을 스펙에 반영
5. **프레젠터 가이드 UI 추가:** 발표자가 시나리오 순서와 다음 액션을 화면에서 확인할 수 있는 안내 패널
6. **시나리오 전환 UX 정의:** 스텝 위저드 패턴으로 전환 흐름을 명확히 정의

#### 권장 조건 (Should)

7. **성능 대시보드 상세 설계:** SVG 게이지, 미니 차트, 목표치 대비 색상 변화 등 구체적 UI 컴포넌트 정의
8. **음성 경고 구현 방식 확정:** 사전 녹음 MP3 (한국어) + Web Audio API. TTS는 네트워크 의존성이 있으므로 비추
9. **반응형 레이아웃:** 1024px ~ 1920px 대응
10. **데모 리셋 기능:** 원클릭으로 모든 상태 초기화

### 종합 평가

이 Requirement Brief는 PoC 데모의 기술적 범위를 잘 정의하고 있으며, 4개 도메인과 시나리오의 매핑이 명확하다. 기존 코드 분석과 Gap 식별도 정직하게 수행되었다. 주요 보완 포인트는 **UX 완성도**와 **데모 안정성 설계**이다. 규제 심사위원은 "기술이 동작하는가"보다 **"이 기술이 아이들을 안전하게 지킬 수 있는가"를 직관적으로 느끼는가**로 판단한다. 현재 가안의 Flask MJPEG + CV2 텍스트 오버레이 방식으로는 그 수준의 시각적 설득력을 달성하기 어렵다.

Flask + Socket.IO + Canvas 아키텍처로 전환하고, 위에서 제안한 UX 설계를 반영하면 규제 심사 수준의 데모가 충분히 달성 가능하다.
