### R-01. 운전자 자격 검증 시스템 부재 [CRITICAL]

**관련 법 조항:**
- 도로교통법 제52조 제2항: 어린이 통학버스 운전자는 해당 어린이교육시설에서 근무하는 사람이어야 함
- 아동복지법 제29조의3: 아동관련기관 종사자 범죄경력 조회 의무
- 도로교통법 시행규칙 제35조: 운전자 안전교육 이수 의무 (매년 3시간)

**현재 코드 상태:**
- `User` 모델(`backend/app/modules/auth/models.py`)에 운전자(DRIVER)와 관련된 자격 정보 필드가 전혀 없음
- 운전면허 번호, 면허 종류, 면허 유효기간 필드 없음
- 범죄경력 조회 결과 필드 없음
- 안전교육 이수 여부/일자 필드 없음
- 운전 경력 연수 필드 없음

```
# User 모델 현재 상태 (auth/models.py:20-43)
class User(Base):
    id, role, phone, email, name, kakao_id, fcm_token, is_active, created_at, updated_at, deleted_at
    # 운전면허, 범죄경력, 교육이수 등 자격 정보 전무
```

**필요한 조치:**
1. Driver 프로필 확장 모델 생성 필요:
   - `license_number` (면허번호, 암호화 저장)
   - `license_type` (면허 종류: 1종 대형 등)
   - `license_expiry` (면허 유효기간)
   - `criminal_check_date` (범죄경력 조회일)
   - `criminal_check_result` (조회 결과)
   - `safety_training_date` (안전교육 이수일)
   - `safety_training_expiry` (교육 만료일)
   - `driving_experience_years` (운전 경력)
2. 차량 배정(VehicleAssignment) 시 자격 미충족 운전자 배정 차단 로직
3. 자격 만료 알림 시스템

**규제 샌드박스 심사 예상 질문:**
> Q: "운전자가 범죄경력이 있는 경우 어떻게 차단합니까?"
> A: (현재 답변 불가 - 시스템에 범죄경력 조회 필드 자체가 없음)

### R-02. 차량 기준 관리 필드 미비 [CRITICAL]

**관련 법 조항:**
- 도로교통법 제52조: 어린이 통학버스 차량 요건 (9인승 이상, 황색 도색 등)
- 자동차관리법 시행규칙 제3조: 차량 정기검사 주기
- SRS 문서 자체 기준: "차량 연식 3년 이내"

**현재 코드 상태:**
- `Vehicle` 모델(`backend/app/modules/vehicle_telemetry/models.py:10-28`)에 있는 필드:
  - `license_plate`, `capacity`, `operator_name`, `insurance_expiry`, `registration_expiry`, `safety_inspection_expiry`, `is_active`
- **누락된 필수 필드:**
  - `manufacture_year` (제조연도) - SRS에서 "3년 이내" 요구했으나 관리 불가
  - `model_name` (차량 모델명)
  - `vehicle_type` (차량 종류: 승합 등)
  - `school_bus_registration_number` (어린이 통학버스 신고번호)
  - `school_bus_registration_date` (신고일)
  - `yellow_painting_confirmed` (황색 도색 확인)
  - `child_protection_sign` (어린이보호표지 부착 확인)

**필요한 조치:**
1. 차량 모델에 연식/신고번호 등 법정 필수 필드 추가
2. 차량 등록 시 통학버스 신고 증빙 문서 업로드 강제화
3. 연식 3년 초과 차량 자동 경고/차단 로직
4. 보험/검사 만료 시 차량 자동 비활성화 로직 (현재 `insurance_expiry` 필드는 존재하나 만료 시 차단하는 로직은 없음)

**규제 샌드박스 심사 예상 질문:**
> Q: "차량 연식 3년 초과 시 어떻게 관리합니까? 신고번호 없는 차량이 배차될 가능성은?"
> A: (현재 답변 불가 - 연식 필드가 없고 신고번호 관리가 없음)

### R-03. 사고 발생 시 책임 소재 명확화 부족 [HIGH]

**관련 법 조항:**
- 여객자동차운수사업법 제81조: 자가용 유상운송 금지
- 민법 제756조: 사용자 책임
- SRS 섹션 4.1: "플랫폼은 '계약의 중개자' 지위"

**현재 코드 상태:**
- `Contract` 모델(`compliance/models.py:57-73`)은 학원-운송사업자 간 계약 구조를 정의하지만:
  - `contract_type`이 단순 문자열 ("charter_transport")
  - 책임 소재 구분 필드 없음 (플랫폼 vs 학원 vs 기사)
  - 보험 가입 증빙 연결 없음
  - 사고 발생 시 프로세스 정의 없음

- 이용약관(`site/src/pages/Terms.tsx`)의 제9조 면책조항이 지나치게 포괄적:
  - "불가항력으로 인한 서비스 중단에 대해 책임을 지지 않습니다"
  - 아동 안전사고에 대한 구체적 책임 분담 조항 없음

**필요한 조치:**
1. Contract 모델에 책임 분담 구조 명시 필드 추가
2. 사고 리포트 모델/모듈 신설 (사고 유형, 관련자, 경위, 후속조치)
3. 이용약관에 아동 안전사고 시 구체적 책임 범위 명시
4. 운송사업자의 보험 증빙과 Contract 연결

**규제 샌드박스 심사 예상 질문:**
> Q: "플랫폼을 통해 배차된 차량에서 아동 안전사고 발생 시, 플랫폼-학원-기사 간 책임 소재가 어떻게 구분됩니까?"
> 준비 답변: "플랫폼은 정보통신융합법상 중개 플랫폼 지위이며, 학원장과 전세버스 사업자 간 직접 운송계약(Contract 테이블)이 체결됩니다. 운송 중 사고의 1차 책임은 계약 당사자인 전세버스 사업자에게, 관리 감독 책임은 학원장에게 귀속됩니다. 다만 플랫폼은 운전자 자격 검증, 차량 기준 확인, 안전도우미 배치 확인 등 시스템적 안전장치를 운영합니다." (단, 현재 코드에서 이 답변의 근거가 되는 기능이 미구현 상태)

### R-04. 어린이 통학버스 신고 의무 관리 부재 [CRITICAL]

**관련 법 조항:**
- 도로교통법 제52조 제1항: "어린이교육시설 등의 설립자/운영자는 ... 관할 경찰서장에게 신고"
- 도로교통법 제52조 제3항: 신고 요건 미충족 시 신고 불수리/취소

**현재 코드 상태:**
- `ComplianceDocument` 모델의 `DocumentType` enum:
  ```python
  class DocumentType(enum.StrEnum):
      INSURANCE_CERT = "insurance_cert"
      POLICE_REPORT = "police_report"    # 이것은 범죄경력 조회서를 의미
      SAFETY_TRAINING = "safety_training"
      VEHICLE_INSPECTION = "vehicle_inspection"
      OTHER = "other"
  ```
- **어린이 통학버스 신고필증**에 대한 전용 document type이 없음
- 신고 상태 관리(미신고/신고완료/신고취소) 추적 불가
- 미신고 차량의 운행 차단 로직 없음

**필요한 조치:**
1. `DocumentType`에 `SCHOOL_BUS_REGISTRATION = "school_bus_registration"` 추가
2. Vehicle 모델에 `school_bus_reg_status` 필드 추가
3. 미신고 차량 배차 차단 비즈니스 로직 구현
4. 신고필증 만료/취소 시 자동 운행 중단 로직

### R-05. 동승보호자 배치 강제 검증 미흡 [HIGH]

**관련 법 조항:**
- 도로교통법 제53조의3 (세림이법): 어린이 통학버스 운행 시 동승보호자 탑승 의무
- SRS 섹션 4.3: "동승 보호자(안전요원) 탑승 의무화 ... 긱 경제 모델 연동"

**현재 코드 상태:**
- `VehicleAssignment` 모델에 `safety_escort_id` 필드가 있음 (nullable) — `vehicle_telemetry/models.py:46-48`
- `EscortShift` 모델이 존재하며 체크인/체크아웃 추적 가능 — `escort/models.py:46-73`
- **그러나:** `safety_escort_id`가 nullable이므로, 동승보호자 없이도 차량 배정 가능
- 동승보호자 미배치 상태에서의 운행 시작 차단 로직 없음
- 동승보호자 체크인 전 출발 방지 로직 없음

**필요한 조치:**
1. VehicleAssignment 생성 시 safety_escort_id 필수화 (또는 미배치 시 경고 플래그)
2. 운행 시작(GPS 수신 시작) 전 동승보호자 체크인 확인 로직
3. 동승보호자 미탑승 상태 운행 시 관제센터 실시간 알림
4. 세림이법 준수 여부 일일 리포트 자동 생성

**규제 샌드박스 심사 예상 질문:**

> A: (현재 답변 곤란 - safety_escort_id가 nullable이고 강제 검증 로직 없음)

### R-06. 전세버스 계약 구조 증빙 불충분 [HIGH]

**관련 법 조항:**
- 여객자동차운수사업법 제81조: 자가용 유상운송 금지
- SRS 섹션 4.1: "학원장과 전세버스 사업자 간의 N:N 전자 계약을 실시간으로 성사시키는 법률적 중개 플랫폼"

**현재 코드 상태:**
- `Contract` 모델은 존재하나 (`compliance/models.py:57-73`):
  - `operator_name`은 단순 문자열 — 전세버스 사업자의 사업자등록번호, 운송사업 면허번호 등 법적 요건 미확인
  - `signed_at`이 nullable — 서명 없는 계약도 유효
  - 전자서명/전자문서 법적 효력 확보 장치 없음
  - 계약서 원본 파일 연결 없음

**필요한 조치:**
1. 전세버스 사업자 정보 확장: 사업자등록번호, 운송면허번호, 대표자명
2. 전자계약 서명 프로세스 구현 (전자서명법 준거)
3. 계약 원본 문서 저장 및 위변조 방지 (해시 검증)
4. 계약 만료 시 자동 운행 중단

### R-07. 실증특례 조건 이행 모니터링 체계 부재 [MEDIUM]

**관련 법 조항:**
- 정보통신융합법 제38조의2 제7항: 실증특례 사업자는 이행 조건을 준수하고 정기 보고 의무

**현재 코드 상태:**
- 실증특례 조건 이행 현황을 추적하는 대시보드나 리포트 모듈 없음
- 운행 건수, 사고 건수, 안전 위반 건수 등 규제 기관 보고용 통계 자동화 없음

**필요한 조치:**
1. 규제 보고용 통계 대시보드 모듈 (월간/분기 자동 리포트)
2. 안전 사고/위반 이벤트 로깅 및 통계
3. 실증특례 조건별 이행 현황 체크리스트 관리

### P-01. 14세 미만 아동 법정대리인 동의 프로세스 불완전 [CRITICAL]

**관련 법 조항:**
- 개인정보보호법 제22조의2: "만 14세 미만 아동의 개인정보를 처리하기 위하여 ... 법정대리인의 동의를 받아야 한다"
- 동법 시행령 제17조: 법정대리인 동의 확인 방법 (서면, 전화, 본인인증 등)

**현재 코드 상태:**
- `GuardianConsent` 모델 존재 (`compliance/models.py:19-34`)
- `consent_method` 필드가 있으며 기본값은 `"phone_otp"`
- **문제점:**
  1. **법정대리인 자격 검증 없음**: `guardian_id`가 실제 해당 아동의 법정대리인인지 확인하는 로직 없음. 단순히 PARENT 역할의 사용자가 임의의 child_id에 동의 가능
  2. **동의 항목 세분화 부족**: `consent_scope`가 자유 형식 JSON — 법정 필수 동의와 선택 동의의 구분이 스키마 레벨에서 강제되지 않음
  3. **동의 거부 시 서비스 제한 범위 미정의**: 위치추적 동의 거부 시 어떤 서비스가 제한되는지 코드에 정의 없음
  4. **법정대리인 본인확인 수단 미흡**: OTP만으로는 법정대리인 본인확인으로 부족할 수 있음 (시행령 제17조는 '공인인증서, I-PIN, 본인확인기관을 통한 확인' 등을 요구)

```python
# compliance/schemas.py:7-17 — 동의 요청 스키마
class ConsentCreateRequest(BaseModel):
    child_id: uuid.UUID
    consent_scope: dict = Field(
        default_factory=lambda: {
            "location_tracking": True,      # 기본 True — 법적 문제 소지
            "push_notification": True,
            "facial_recognition": False,
        },
    )
    consent_method: str = Field(default="phone_otp")
```

**법적 위험:**
- `consent_scope`의 기본값이 `location_tracking: True` — 사용자가 명시적으로 동의하지 않아도 위치추적이 켜지는 구조. 이는 개인정보보호법 제22조 제4항 "개인정보처리자는 정보주체에게 ... 동의 없이 처리할 수 있는 개인정보와 필수 동의 사항을 구분하여 ... 명확하게 인지할 수 있도록" 위반 소지

**필요한 조치:**
1. 동의 범위를 구조화된 enum/필드로 변경 (필수/선택 구분)
2. 기본값 제거 — 사용자의 명시적 선택 필수화
3. 법정대리인 본인확인 수단 강화 (CI 연동 또는 공동인증서)
4. 법정대리인-아동 관계 증빙 프로세스 (가족관계증명서 등)
5. 동의서 전문(全文) 열람 기록 저장

- 위치정보법 제24조: "개인위치정보의 수집·이용·제공 사실 확인자료를 ... 통신사실 확인자료로서 6개월간 보관"
- 위치정보법 제24조의2: "위치정보 보존 기간 경과 시 지체 없이 파기"

**현재 코드 상태:**
- `GpsHistory` 테이블에 차량 위치 이력이 무기한 저장 (`vehicle_telemetry/models.py:58-76`)
- Redis 캐시의 GPS 데이터 TTL은 300초 (`config.py:72`)로 설정되어 있으나 이는 캐시일 뿐

**필요한 조치:**
1. GPS 이력 데이터 보존기간 정의 (위치정보법 6개월 의무보관 + 이후 파기)
